Concurrency in Low-Level Design

How to Handle Concurrency in LLD Interviews — Correctness, Coordination, Scarcity

LLD Interview Prep

Overview

Three main categories of concurrency problems in low-level design: Correctness, Coordination, and Scarcity. Use concurrency primitives (locks, atomics, blocking queues, semaphores) to solve them.

Categories of Concurrency Issues

CategoryDescriptionCommon ProblemsSolutions
Correctness Shared state corruption due to simultaneous thread access Check-then-act race, read-modify-write race Locks (mutex, synchronized), atomic variables
Coordination Work flow between threads (producer-consumer) Busy-waiting, unbounded queue growth Blocking queues (bounded), back pressure
Scarcity Limiting access to finite resources API rate limits, connection pool overload Semaphores, blocking queues as resource pools

1. Correctness

Definition

Multiple threads access and modify shared state without synchronization → inconsistent or corrupted data.

Common Patterns

  • Check then Act: Thread checks a condition (e.g., seat available), then acts (books seat). Another thread can intervene between check and act.
  • Read-Modify-Write: Read value, modify (e.g., increment), write back. Simultaneous updates can lose increments.

Read-modify-write — without protection (the bug)

count++ looks like one step, but it is read → add 1 → write. Another thread can run between those steps, so updates can be lost.

// Java — THE BUG: read-modify-write without protection
private int count = 0;

public void increment() {
    count++;  // Actually: read count, add 1, write count
}

Read-modify-write — fix with AtomicInteger

incrementAndGet() performs the update in one atomic step (compare-and-swap under the hood), so concurrent increments don’t corrupt the value.

// import java.util.concurrent.atomic.AtomicInteger;

private AtomicInteger count = new AtomicInteger(10);

public void increment() {
    count.incrementAndGet();  // Hardware-level atomic operation
}

Example

Ticket booking: two threads book the same seat; one overwrites the other.

Solutions

  • Locks: Java synchronized, Python Lock — make check and act atomic
  • Atomic variables: Java AtomicInteger for single-variable updates (compare-and-swap). Python lacks built-in atomics; use locks.

When to use atomics vs locks

Use the right tool for how much state must move together:

  • Atomic variables: Use when you are updating one shared value — a counter, flag, or single numeric field — and nothing else has to change in lockstep with it. The hardware/CAS makes that one update atomic.
  • Locks (synchronized / Lock): Use when you must update two or more pieces of state (or a structure plus an invariant) so they stay consistent with each other. No single atomic type can tie them together; one critical section ensures all observers see a valid intermediate-free snapshot.

Rule of thumb: one independent variable → atomics; multi-field or check-then-act on shared data → lock around the whole logical operation.

Code — lock covers check and act

The lock must wrap both the condition check and the state change so another thread cannot book the same seat between isAvailable() and setOccupant(). (lock and seat would be fields on the booking service / seat holder.)

// THE FIX: lock covers both check AND action
public boolean bookSeat(String seatId, String visitorId) {
    synchronized (lock) {
        if (seat.isAvailable()) {
            seat.setOccupant(visitorId);
            return true;
        }
        return false;
    }
}

Key insight: Make the entire operation atomic to prevent interference.

2. Coordination

Definition

Communication and synchronization between threads — passing work from producers to consumers.

Scenario

Background task queue for sending emails asynchronously without blocking the main request thread.

Problems

  • Busy-waiting: Worker threads poll the queue inefficiently, wasting CPU
  • Unbounded growth: Work arrives faster than consumers process → memory overflow

Naive Solutions

Busy-wait wastes CPU; adding sleep reduces waste but adds latency.

Proper Solution

  • Blocking queues: Consumers sleep when no work; wake when work arrives
  • Bounded blocking queues: Limit size; apply back pressure — block producers when full

Implementation

  • Java: BlockingQueue — put(), take()
  • Python: queue module — blocking by default, put(), get()

Key insight: Coordination = work handoff. Bounded blocking queues handle idle wait and overload.

3. Scarcity

Definition

Managing access to limited resources — API rate limits, database connection pools.

Example

Limit concurrent API requests to 10 when 50 threads may call simultaneously.

Tools

  • Semaphores: Bucket of permits; threads acquire to proceed, release after use
  • Blocking queues as resource pools: Store actual resources (e.g., DB connections) in a bounded blocking queue; check out and return

Important

Always release permits/resources even on exceptions — use try-finally to avoid deadlock or starvation.

Implementation

  • Java: Semaphore — acquire(), release()
  • Python: Semaphore — acquire(), release()

Key insight: Careful resource management; ensure release to avoid deadlock and ensure fairness.

Mental Checklist for Interview

  • Shared mutable state accessed by multiple threads? → Correctness (locks, atomics)
  • Work flowing asynchronously (producer-consumer)? → Coordination (blocking queues, back pressure)
  • Fixed limit on resources threads must share? → Scarcity (semaphores, resource pools)

Key Takeaways

  • Concurrency bugs stem from timing and ordering between threads
  • Correctness: unsynchronized shared state → atomic operations
  • Coordination: communication and task scheduling → blocking queues
  • Scarcity: finite resources → semaphores; ensure proper release
  • Fundamental primitives: locks, atomics, blocking queues, semaphores
  • Consider exception safety for scarce resources and locks
