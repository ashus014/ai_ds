Concurrency in Low-Level Design

How to Handle Concurrency in LLD Interviews — Correctness, Coordination, Scarcity

📅 March 20, 2026

Overview

Three main categories of concurrency problems in low-level design: Correctness, Coordination, and Scarcity. Use concurrency primitives (locks, atomics, blocking queues, semaphores) to solve them.

Categories of Concurrency Issues

CategoryDescriptionCommon ProblemsSolutions
Correctness Shared state corruption due to simultaneous thread access Check-then-act race, read-modify-write race Locks (mutex, synchronized), atomic variables
Coordination Work flow between threads (producer-consumer) Busy-waiting, unbounded queue growth Blocking queues (bounded), back pressure
Scarcity Limiting access to finite resources API rate limits, connection pool overload Semaphores, blocking queues as resource pools

1. Correctness

Definition

Multiple threads access and modify shared state without synchronization → inconsistent or corrupted data.

Common Patterns

  • Check then Act: Thread checks a condition (e.g., seat available), then acts (books seat). Another thread can intervene between check and act.
  • Read-Modify-Write: Read value, modify (e.g., increment), write back. Simultaneous updates can lose increments.

Read-modify-write — without protection (the bug)

count++ looks like one step, but it is read → add 1 → write. Another thread can run between those steps, so updates can be lost.

// Java — THE BUG: read-modify-write without protection
private int count = 0;

public void increment() {
    count++;  // Actually: read count, add 1, write count
}

Read-modify-write — fix with AtomicInteger

incrementAndGet() performs the update in one atomic step (compare-and-swap under the hood), so concurrent increments don’t corrupt the value.

// import java.util.concurrent.atomic.AtomicInteger;

private AtomicInteger count = new AtomicInteger(10);

public void increment() {
    count.incrementAndGet();  // Hardware-level atomic operation
}

Example

Ticket booking: two threads book the same seat; one overwrites the other.

Solutions

  • Locks: Java synchronized, Python Lock — make check and act atomic
  • Atomic variables: Java AtomicInteger for single-variable updates (compare-and-swap). Python lacks built-in atomics; use locks.

When to use atomics vs locks

Use the right tool for how much state must move together:

  • Atomic variables: Use when you are updating one shared value — a counter, flag, or single numeric field — and nothing else has to change in lockstep with it. The hardware/CAS makes that one update atomic.
  • Locks (synchronized / Lock): Use when you must update two or more pieces of state (or a structure plus an invariant) so they stay consistent with each other. No single atomic type can tie them together; one critical section ensures all observers see a valid intermediate-free snapshot.

Rule of thumb: one independent variable → atomics; multi-field or check-then-act on shared data → lock around the whole logical operation.

Code — lock covers check and act

The lock must wrap both the condition check and the state change so another thread cannot book the same seat between isAvailable() and setOccupant(). (lock and seat would be fields on the booking service / seat holder.)

// THE FIX: lock covers both check AND action
public boolean bookSeat(String seatId, String visitorId) {
    synchronized (lock) {
        if (seat.isAvailable()) {
            seat.setOccupant(visitorId);
            return true;
        }
        return false;
    }
}

Key insight: Make the entire operation atomic to prevent interference.

2. Coordination

Definition

Communication and synchronization between threads — passing work from producers to consumers.

Scenario

Example: several API request threads enqueue background work (e.g. sendEmail tasks), and worker threads pull from a shared queue and process them — classic producer–consumer.

Design questions

  • How does a consumer know work arrived? If it spins checking the queue, it burns CPU. If it sleeps on a fixed interval, it adds latency and still isn’t ideal.
  • What if producers are faster than consumers? An unbounded queue can grow until you run out of memory — you need back pressure (e.g. a bounded queue that blocks producers when full).

Approach 1 — busy-waiting (wastes CPU)

The worker loop never blocks: it keeps asking “is the queue empty?” as fast as the CPU allows. When there is no work, you get 100% useless spinning.

// BAD: busy-waiting burns CPU
while (true) {
    if (!queue.isEmpty()) {
        Task task = queue.poll();
        process(task);
    }
    // spins forever when queue is empty
}

Approach 2 — sleep, then poll (saves CPU, adds latency)

Sleeping between checks avoids burning the core, but a task that arrives right after a check may wait up to the full sleep interval — you’ve traded CPU for artificial delay.

// ALSO BAD: sleep-polling adds latency
while (true) {
    if (!queue.isEmpty()) {
        Task task = queue.poll();
        process(task);
    } else {
        Thread.sleep(100); // e.g. up to ~100ms before noticing new work
    }
}

What we want — block until work is there

Workers should sleep with zero busy CPU when the queue is empty and wake as soon as a producer adds a task. A blocking queue does that: take() / get() park the consumer until an item appears. Most mainstream runtimes ship this — Java java.util.concurrent.BlockingQueue, Python queue.Queue (blocking put/get), C# BlockingCollection, Go channels, etc.

Next problem — work arrives faster than consumers can drain

If producers keep enqueueing and consumers cannot keep up, an unbounded queue grows until you run out of memory. The fix is a bounded blocking queue: you set a maximum capacity (e.g. 1000 tasks). When the queue is full, put() blocks — producers wait until consumers free a slot. That slowdown is intentional: it is back pressure propagating upstream so the system naturally slows down instead of buffering an infinite backlog. Together, blocking take() (empty queue) and blocking put() (full queue) give you both idle efficiency and overload control.

Example — bounded blocking queue (Java)

LinkedBlockingQueue(capacity) is bounded; put / take are the blocking entry points.

// Bounded queue — blocks when full (backpressure)
BlockingQueue<Task> queue = new LinkedBlockingQueue<>(1000);

// Producer — API handler
public void handleSignup(User user) {
    saveUser(user);
    queue.put(new EmailTask(user));  // blocks if queue is full
    return success();
}

// Consumer — worker thread
while (true) {
    Task task = queue.take();  // blocks if queue is empty, zero CPU
    process(task);
}

Implementation cheat sheet

  • Java: BlockingQueue — put(), take(); common impl LinkedBlockingQueue
  • Python: queue.Queue — blocking put(), get() by default

Key insight: Coordination = work handoff. A blocking queue gives “sleep until work” and “wake on arrival”; a bounded blocking queue also answers “too much work” with back pressure instead of unbounded memory growth.

3. Scarcity

Definition

Managing access to a finite resource — only so many callers may use it at once (concurrency cap), analogous to rate limits or fixed-size pools.

Example

Fifty threads might call a service, but the service (or an upstream API) allows only 10 concurrent operations. You need a gate that lets at most 10 through; the other 40 wait their turn instead of all hammering at once.

Semaphore — bucket of permits

A semaphore is a fixed-size “bucket” of permits. Before doing the guarded work, a thread acquires one permit (takes it out of the bucket). When the work finishes, it releases the permit (puts it back). If a thread tries to acquire() when the bucket is empty, it blocks until another thread release()s — same idea as waiting for a free slot.

Code — naive release (permit leak on exception)

If the work throws, release() never runs: permits leak out of the bucket until none are left and every thread blocks forever.

// import java.util.concurrent.Semaphore;

Semaphore permits = new Semaphore(10); // max 10 concurrent

public void download(String url) {
    permits.acquire(); // blocks if no permit free
    doDownload(url);   // if this throws → release() never runs ❌
    permits.release();
}

Code — try / finally (always put the permit back)

Wrap the work in try and put release() in finally so it runs whether doDownload succeeds or throws. (Use try-with-resources or small helpers if you prefer; the interview answer is “guaranteed release.”)

Semaphore permits = new Semaphore(10); // max 10 concurrent

public void download(String url) {
    permits.acquire();
    try {
        doDownload(url);
    } finally {
        permits.release(); // ALWAYS runs, even if exception thrown
    }
}

Pooling stateful objects — not just a counter

Sometimes scarcity isn’t only “how many requests at once” — you are managing real objects that have state and cost (TCP handshakes, auth, memory). Example: database connections. Opening a new connection per call is slow and can exhaust the DB; instead you keep a fixed set (e.g. 10) open and reuse them.

Pattern: put those live objects in a blocking queue. Threads take() a connection, run queries, then put() it back. If all connections are checked out, the next caller blocks on take() until someone returns one — same “finite resource” idea as a semaphore, but the queue holds the actual pooled instances.

// Initialize pool with connections (example API)
BlockingQueue<Connection> pool = new LinkedBlockingQueue<>(1000);
for (int i = 0; i < 10; i++) {
    pool.put(createConnection());
}

// Usage — borrow, use, always return
public Result query(String sql) {
    Connection conn = pool.take(); // blocks if pool is empty
    try {
        return conn.execute(sql);
    } finally {
        pool.put(conn); // ALWAYS return to pool, even on exception
    }
}

createConnection() / conn.execute stand in for your driver APIs; production pools add health checks, timeouts, and max wait — the interview shape is take → try → finally put.

APIs

  • Java: Semaphore — acquire() / release(); pools — BlockingQueue + take() / put()
  • Python: threading.Semaphore or asyncio.Semaphore; pools often use queue.Queue or a library pool

Key insight: Scarcity = finite concurrency or finite expensive objects. Semaphores count permits; a blocking queue pool stores the real resources. Always return permits or connections in finally so nothing leaks.

Mental Checklist for Interview

  • Shared mutable state accessed by multiple threads? → Correctness (locks, atomics)
  • Work flowing asynchronously (producer-consumer)? → Coordination (blocking queues, back pressure)
  • Fixed limit on resources threads must share? → Scarcity (semaphores, blocking queues for pools)

In conclusion

  • Correctness (shared state) — protect shared mutable state with locks (e.g. synchronized, explicit locks) and atomic variables when a single field’s update must be indivisible.
  • Coordination (work handoff) — hand off work between producers and consumers with blocking queues (block when empty, optionally block when full for back pressure).
  • Scarcity (limited resources) — cap concurrent use or pool expensive objects with semaphores (permits) and blocking queues (actual pooled resources like connections); always release / return in finally.

Key Takeaways

  • Concurrency bugs stem from timing and ordering between threads
  • Correctness: unsynchronized shared state → locks + atomics (pick by how much state moves together)
  • Coordination: producer–consumer and back pressure → blocking queues
  • Scarcity: finite slots or pooled objects → semaphores and blocking-queue pools; exception-safe release
  • Fundamental primitives: locks, atomics, blocking queues, semaphores
  • Consider exception safety for scarce resources and locks
