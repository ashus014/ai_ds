Understanding Kubernetes

From bare metal and VMs to containers and orchestration — written for students new to cloud and distributed systems.

📅 Notes — March 20, 2026

What this page is for

This is a structured summary of a Kubernetes introduction: why it exists, how deployment evolved, what containers are, and how a cluster is put together. If you have never touched Kubernetes, read in order — each section builds on the last.

Prerequisite tip: Comfort with Docker (build an image, run a container) makes Kubernetes much easier, because Kubernetes orchestrates containers — it does not replace knowing what a container is.

Key topics covered

  • Traditional deployment challenges
  • Cloud computing and cloud-native technologies

  • Kubernetes architecture and main components

How we used to deploy software

Early on, teams built apps on laptops and deployed to physical servers (bare metal) with static IPs so the public internet could reach them. That worked, but:

  • Hardware is expensive; you need space, power, cooling, and people to rack and replace machines.
  • Scaling meant buying bigger CPUs or more RAM — slow Capex, not something you do in minutes.
  • There is no elastic “turn down capacity overnight” — you mostly pay for peak.

Cloud (e.g. AWS) changed the model: you rent virtual machines (VMs) by the hour. You stop managing physical boxes, and you can add instances when traffic spikes. Providers also added managed building blocks — databases, load balancers, CDNs — so you compose systems instead of operating every layer yourself.

Beginner snag: Even on cloud, “my laptop” and “the server” often differed in OS packages, library versions, and config — so reproducing production locally stayed painful until containers became mainstream.

Containers vs virtual machines

A VM image often bundles a full guest operating system per app. That is reliable isolation, but images are heavy (gigabytes), boot is slower, and you pay RAM/CPU for many duplicate OS kernels.

Containerization packages your app + dependencies into a lightweight image. At runtime, containers on the same host typically share the host’s Linux kernel but get isolated process, filesystem, and network views. Result: faster start, denser packing, and the same artifact runs in dev, staging, and prod if the kernel ABI is compatible.

IdeaVirtual machineContainer
Isolation unit Full OS per VM Processes + namespaces/cgroups on one OS
Typical image size Large (full OS) Smaller (app layers)
“Works on my machine” Still config drift between envs Same image → more predictable

Docker made building and running containers approachable; today you also hear containerd, CRI-O, etc. — runtimes that actually start containers while Kubernetes speaks to them through a standard interface.

Why orchestration exists

Each container must be started, stopped, restarted, placed on a machine with enough CPU/RAM, wired to the network, and observed (logs, health). When you have one container, scripts are fine. When you have dozens of services each scaled to many replicas, manual ops break down.

Concrete scenario: Your API needs 5 copies for traffic; a worker needs 2; a cache sidecar rides next to the API. One VM dies — which processes move where? Who recreates them? Who attaches the same secrets and service DNS names? Orchestration automates those decisions.
  • Scale replicas up/down with load or policy
  • Replace crashed containers (self-healing)
  • Roll out new versions with controlled updates
  • Attach storage, secrets, and networking consistently

Kubernetes: origin and role

Google ran internal workloads at huge scale with Borg. Lessons from Borg fed into Kubernetes (k8s — “k” + 8 letters + “s”), an open-source orchestrator designed from the ground up for the container era.

In 2014, Kubernetes was donated to the Cloud Native Computing Foundation (CNCF), which helped grow a neutral ecosystem around it.

What Kubernetes does (one sentence): You declare a desired state (how many pods, what image, how exposed); the system continuously drives the actual state toward that goal.

  • Groups containers into pods — the smallest deployable unit (often one main container; sometimes sidecars like log shippers or proxies).
  • Is cloud-agnostic: same core ideas on AWS, GCP, Azure, or on-premises — reducing vendor lock-in for the orchestration layer (you still choose where nodes live).

Architecture — control plane

The control plane is the “brain” — usually replicated for availability. It accepts API calls, stores intent, schedules work, and runs controllers that reconcile state.

ComponentRole (beginner view)
API Server Front door for the cluster. kubectl and controllers talk to it. Validates and persists objects (Pods, Services, etc.).
Controller Manager Runs control loops: “should 3 replicas exist?” If a pod vanished, create another. Many small controllers, one binary in practice.
Scheduler Picks a worker node for each new pod based on resources, constraints, and policies.
etcd Distributed key-value store holding cluster state. Lose etcd carelessly and you lose the source of truth (it is backed up in real operations).

Architecture — worker nodes

Worker nodes run your workloads. The control plane decides what should run; nodes make it real.

ComponentRole (beginner view)
Kubelet Agent on each node. Talks to API Server, starts/stops containers via the runtime, reports health.
Kube-proxy Maintains network rules so Service IPs and ports reach the right pods (often via iptables/IPVS or dataplane-specific paths).
Container runtime (CRI) Actually runs images — e.g. containerd, CRI-O. Kubernetes uses the Container Runtime Interface so the core stays runtime-neutral.

How reconciliation works

  1. You apply YAML (or an API call): “I want 3 replicas of this app.”
  2. API Server stores that in etcd.
  3. Scheduler assigns pending pods to nodes.
  4. Kubelet pulls the image if needed and starts containers.
  5. Controllers watch state: if a pod dies, a new one is created — self-healing.
Analogy: A thermostat’s desired temperature vs the current room temperature. Kubernetes repeatedly closes the gap — not magic, just disciplined automation with clear APIs.

Kube-proxy (with Services) helps traffic reach pods even as pod IPs change during restarts — important for stable client access and later load balancer integration.

Cloud Controller Manager (CCM)

Kubernetes core stays generic; cloud-specific integrations (e.g. creating a cloud load balancer, attaching a disk from AWS/GCP) live in the Cloud Controller Manager. It calls provider APIs so your Service of type LoadBalancer can materialize an actual ELB/GCP LB — without baking AWS-only code into every cluster.

Why teams adopt Kubernetes

  • Scale and availability: Spread work across nodes; replace failed pieces automatically.
  • Portability: Same workload descriptions across clouds and on-prem (with caveats for storage/LB integrations).
  • Ecosystem: Huge community, CNCF projects, tooling, and hiring market around k8s.

Trade-off: operational complexity is real — you learn primitives (Pods, Deployments, Services, ConfigMaps, Ingress, etc.) and often need a platform team or managed offering (EKS, GKE, AKS) to run it safely in production.

Closing summary

Kubernetes is the dominant open-source container orchestration platform. It tackles deployment, scaling, and reliability for distributed apps by keeping desired state in etcd and reconciling reality on worker nodes — with cloud hooks via CCM where needed.

For a new engineer: learn containers first, then map each YAML resource to “which component enforces this?” (API Server, scheduler, kubelet, controllers). That mental model beats memorizing every flag.
