Singleton Design Pattern — From Basics to Spring Boot Reality

Thread safety, double-checked locking, volatile, and when the framework makes it unnecessary

📅 Notes — March 20, 2026

Introduction

When you start learning system design or backend engineering, one pattern keeps coming up again and again: Singleton.

At first, it looks simple. Once you go deeper — threading, JVM, Spring — it becomes very interesting.

Here’s a step-by-step path from beginner ideas to real-world usage.

What is Singleton?

Singleton is a design pattern that ensures:

  • Only one instance of a class exists
  • That instance is globally accessible

In simple terms: one object is created, and the same reference is used everywhere. (Hindi shorthand you might use: “Ek hi object banega, aur wahi har jagah use hoga.”)

Why do we need Singleton?

Imagine you are building a backend system. Some components should not have multiple instances:

  • Database connection manager
  • Logger
  • Configuration manager
  • Cache

If multiple objects are created by mistake, you risk data inconsistency, memory waste, and race conditions.

Basic Singleton implementation

public class Browser {

    private static Browser browser;

    private Browser() {}

    public static Browser getInstance() {
        if (browser == null) {
            browser = new Browser();
        }
        return browser;
    }
}

What’s happening?

  • Constructor is private → callers cannot instantiate directly
  • A static field holds the single instance
  • getInstance() is the controlled access point

Problem: not thread-safe

Under multithreading, two threads can both see null, both enter the block, and both create an instance — the pattern breaks.

Fix 1: synchronized method

public static synchronized Browser getInstance() {
    if (browser == null) {
        browser = new Browser();
    }
    return browser;
}

Pros: thread-safe, easy to implement.

Cons: every call acquires the lock → can hurt performance when getInstance() is hot.

Fix 2: double-checked locking

public static Browser getInstance() {
    if (browser == null) {
        synchronized (Browser.class) {
            if (browser == null) {
                browser = new Browser();
            }
        }
    }
    return browser;
}

Why “double check”?

  • Check before locking → avoid locking on every call after initialization
  • Check after locking → only one thread constructs the instance

Hidden issue: publication without volatile

browser = new Browser(); is not a single atomic “step” in the memory model sense. The JVM can reorder visible effects so another thread might observe a non-null reference before the object is fully constructed.

Final piece: volatile

private static volatile Browser browser;

volatile helps establish the right happens-before relationship so that the fully constructed object is what other threads see when they read the reference — together with double-checked locking, this is the classic correct lazy Singleton in Java.

Comparison

Approach Thread-safe Performance Notes
Simple (no sync) No Fast Unsafe under concurrency
synchronized method Yes Slower (lock every call) Fine for low call volume
DCL without volatile Risky Better Broken publication possible
DCL + volatile Yes Good Standard lazy Singleton fix in Java

Reality check: Spring Boot

In a typical Spring Boot application, you usually do not hand-roll Singleton for services.

@Service
public class OrderService { }

By default, Spring registers a single bean per scope (default singleton scope) in the application context and injects that same instance wherever it’s needed.

@Autowired OrderService s1;
@Autowired OrderService s2;
// s1 == s2  →  true (same bean)
Aspect Manual Singleton Spring singleton bean
Lifecycle You own it Container manages it
Boilerplate Higher Minimal
Typical recommendation in Spring apps Rarely needed Prefer this

Practical mindset: in many modern backends, prefer letting the framework manage object scope and wiring instead of reinventing Singleton for every service.

Summary

  • Singleton = single shared instance + global access
  • Naive lazy init is not safe under threads
  • synchronized method is safe but can be heavy
  • Double-checked locking + volatile is the usual correct lazy pattern in Java
  • In Spring Boot, default-scoped beans are already singletons — manual pattern often unnecessary

Interview line

“In Java, Singleton can be implemented with double-checked locking and a volatile reference for safe publication under concurrency. In Spring Boot, the default bean scope is singleton, so the container handles that — you rarely need a hand-written Singleton for typical services.”
