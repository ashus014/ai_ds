Apache Kafka

Distributed event streaming — crash-course notes: problems it solves, architecture, consumer groups, and a Node.js-oriented setup path.

Key concepts and problems Kafka solves

Kafka addresses real-time data throughput for systems that must handle very high rates of events, for example:

  • Live location tracking for delivery partners (e.g. Zomato riders).
  • Real-time chat at large concurrency (e.g. Discord).
  • Ride-booking platforms (Ola, Uber) streaming driver and trip updates.

Traditional databases often hit throughput bottlenecks when write/read rates climb (hundreds or thousands of ops per second and beyond), risking failures or severe slowdowns.

Throughput means how many read/write operations per second a system can sustain. Kafka is built for very high throughput; typical databases trade that for durable storage, indexing, and rich querying.

Not a database replacement: Kafka retains data for a limited, configurable window and streams events at scale. Databases excel at long-lived storage and query models — use both where each fits.

Kafka architecture overview

Kafka acts as a distributed message broker. Main pieces:

Click the diagram to open a full-size view; press Esc or click the dimmed backdrop to close.

ComponentDescription
Producer Apps or services that publish messages to Kafka topics.
Topic Logical channel or category producers write to.
Partition Topics are split into partitions for parallelism and scale.
Consumer Subscribers that read messages from topics.
Consumer group Consumers that share partition assignment and split the read load.
Zookeeper Manages cluster metadata, partition assignments, and consumer group coordination internally.
  • Topics group messages logically; partitions split volume and enable parallel consumption.
  • Within one consumer group, each partition is consumed by at most one consumer at a time — that preserves order inside the partition and avoids duplicate processing in the group.
  • Multiple consumer groups can subscribe to the same topic independently (pub/sub across applications).
  • Kafka rebalances partition assignments when consumers join or leave a group.

Real-world use case: food delivery (Zomato)

  • Partner location updates stream into Kafka (e.g. roughly per-second).
  • Messages land in partitions (e.g. by region or custom key).
  • Analytics, customer apps, fare engines, etc. consume the same stream via different consumer groups.
  • Downstream services process in near real time and persist aggregates or snapshots to their own databases.
  • Ingestion stays decoupled from OLTP databases, avoiding overload on a single write path.

Kafka vs traditional databases

FeatureKafkaTraditional DB (e.g. PostgreSQL, MongoDB)
Throughput Very high (millions of events/sec at scale) Lower to moderate for typical transactional workloads
Data retention Time- or size-bounded (streaming log) Durable, long-term storage
Query support Sequential consumption patterns; limited ad-hoc query Indexes, joins, rich queries
Use case Streaming, buffering, event pipelines Transactional state, reporting source of truth
Scalability Horizontal scale via partitions and brokers Often vertical + bounded horizontal patterns

Internal concepts (producer, consumer, Zookeeper)

  • Producers choose topic and partition (explicitly or via key hashing / custom logic).
  • Consumers: one consumer may read many partitions; one partition → one consumer per group; several consumers in a group share partitions; several groups read the same topic independently.
  • Zookeeper is used internally by Kafka for metadata management, leader election, and cluster coordination.

Practical setup and code outline (Node.js)

Assumptions: comfortable with Node.js basics and distributed-systems vocabulary; Docker available.

  1. Run Zookeeper (commonly port 2181).
  2. Run Kafka broker wired to Zookeeper (commonly 9092).
  3. Use a Node Kafka client to: create an admin client and topics with partition counts; implement producers sending JSON (or other payloads) to chosen partitions; implement consumers subscribing asynchronously.

Example sketch: topic rider_updates with two partitions (e.g. North → partition 0, South → partition 1). Producer accepts input and routes by region; consumer logs messages; consumer groups show load spread when you add instances.

Important properties and behaviors

  • Partition assignment: one consumer may own many partitions; each partition is assigned to exactly one consumer in a given group; adding consumers triggers rebalance.
  • Ordering: guaranteed within a partition; not guaranteed across partitions.
  • Replication: configurable replication factor for fault tolerance.
  • Consumer groups: multiple groups on one topic let different services process the same stream without stepping on each other’s offsets.

Benefits and typical use cases

  • High-throughput, fault-tolerant messaging and event streaming.
  • Decouples high-rate ingestion from databases and batch jobs.
  • Common in delivery, rideshare, telecom, banking, and chat-scale systems.
  • Buffers spikes so databases and workers consume at sustainable rates.
  • Partitioning + consumer groups = flexible scale-out consumption.

Closing takeaways

Kafka complements databases — it is not a drop-in substitute. Partitioning and consumer-group mechanics enable scalable real-time pipelines. Hands-on practice (Docker + Node producer/consumer) solidifies the model faster than theory alone.
