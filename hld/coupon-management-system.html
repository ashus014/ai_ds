Coupon Management System

Discount coupon generation, search, lock, redeem — HLD notes

Updated: 21 March 2026

Reference

Discussion prompt / problem framing: LeetCode — Coupon Management System (system design)

Problem Statement

Design a coupon generation system. - System will allow users to create discount coupons, manage these coupons. - Also, system should be able to validate the coupons. - One user should get single coupn per day. - Once a user gets a coupon he/she has to buy something on the website with in 5 mins otherwise coupon will be availbale for other user. - Coupons should not redeemed multiple times. - System should focus on managing the coupons.

Functional Requirements

  • Admin should be able to create 100k coupons daily.
  • User should be able to search coupons.
  • User should be able to get coupon and redeem it.
  • One user should get single coupn per day.
  • Once a user gets a coupon he/she has to buy something on the website with in 5 mins.
  • If user did not use coupon 5 in minute then coupon will be availbale for other user.
  • System should be able to validate the coupons (user can not be redeemed multiple times same coupon).

Clarified intent: one coupon claim per user per day; unused lock expires after ~5 minutes so others can claim; a single-use coupon cannot be redeemed more than once.

Non-Functional Requirements

  • The system should have high availability for searching & viewing coupons, but should be consistent for getting coupons (only one user should get a coupon).
  • The system should have low latency search.
  • The system should be scalable and able to handle high throughput in the form of popular coupons (10M users for 100k coupons).
  • The system is read heavy, and thus needs to be able to support high read throughput (100:1).

Core Entities

1. Coupon: This Entity store essential information about a coupon, including details like the date, description, type. It acts as the central point of information for each unique coupon. 2. Redeem: Represents the essential information about redeem. 3. User: Represents the individual interacting with the system.
  • Coupon — date, description, type, status, ids; central record per unique coupon.
  • Redeem — ties user, coupon, order, time.
  • User — search, lock, redeem.

APIs

1. Create coupon

POST /coupons → returns couponId. Used by admin or merchant. Creates a new coupon.

Request body

POST /coupons { "category": "electronics", "description": "10% off", "expiryDate": "2026-01-31", "type": "PERCENTAGE" }

Response

201 Created { "couponId": "COUP123" }

2. Search coupon

GET /coupons/search?search_term={search_term}&type={type} — Users browse available coupons; returns matches.

Example query

/coupons/search?search_term=laptop&type=PERCENTAGE

Response

[ { "couponId": "COUP123", "description": "10% off laptops", "type": "PERCENTAGE", "category": "electronics" } ]

3. Get (lock) coupon

POST /coupons/locked/{couponId} → returns coupon payload / status.

Locked coupon means: user selects a coupon, system reserves it temporarily, no one else can redeem it during the lock window.

Why POST instead of GET? Locking changes state (side effect), so POST is appropriate.

Flow: User clicks “use coupon”. Backend sets something like:

status = LOCKED lockedByUser = USER123 lockExpiresAt = <timestamp>

Response

{ "couponId": "COUP123", "status": "LOCKED" }

4. Redeem coupon

POST /coupons/redeem/{couponId} — User applies coupon at checkout.

What happens: Validate coupon, verify lock (optional but good), mark REDEEMED, prevent reuse for single-use coupons, persist redemption record.

Request body

POST /coupons/redeem/COUP123 { "userId": "USER123", "orderId": "ORDER999" }

Response

{ "status": "SUCCESS", "discountApplied": 120, "finalPrice": 880 }

Important validations: coupon not expired; belongs to user if personalized; not already redeemed; optionally was locked by same user and lock still valid (lock may have expired before redeem attempt).

High-level architecture (final draft)

Component summary

  • API Gateway — Auth, rate limiting, routing to microservices.
  • Search Service — Coupon discovery; reads from Elasticsearch for low-latency search (not full table scans on SQL).
  • Coupon Generation Service — Create coupons; Redis (global counter) with atomic INCR for unique couponId; persist to DB; index/update Elasticsearch.
  • Coupon Redeem Service — Lock and redeem; Redis (coupon lock) for TTL-based locks; DB for authoritative state; update Elasticsearch after redemption so coupons disappear or update in search.
  • Redis (global counter) — Sequence for IDs: service gets next number, builds id, stores row in DB.
  • Redis (coupon lock) — Per-coupon lock with expiry (e.g. 5 minutes); reduces races vs relying on DB alone under contention.
  • Database — Source of truth for coupons and redemptions.
  • Elasticsearch — Search index; kept in sync with DB (async indexer / events after commit).

High-level design — request flows

1. Admin creates coupon

Admin calls POST /coupons. Coupon Generation Service validates the body, generates couponId (e.g. via Redis INCR), sets initial status = ACTIVE, saves to the database, then updates Elasticsearch so the coupon is searchable.

After creation, a typical DB row includes:

couponId = C123 status = ACTIVE

2. User searches coupon (naive vs optimized)

User searches by term and/or type. Search Service serves this traffic.

Naive approach: Query SQL directly, e.g.

SELECT * FROM coupons WHERE category LIKE '%electronics%'

This becomes slow at scale: large table, heavy LIKE scans, weak scaling for free text. Fine for a first HLD sketch; optimization = proper indexes, cache, or Elasticsearch as in the architecture diagram.

Optimized path: Search Service queries Elasticsearch (inverted index on description, category, type) for fast, filterable search while DB remains transactional source of truth.

3. User selects coupon → lock

After results are shown, user selects a coupon. App calls POST /coupons/locked/{couponId} (same intent as reserving the code).

Backend should ensure only one user holds the reservation: e.g. Redis SET coupon:lock:<id> <userId> NX EX 300, then reflect LOCKED in DB if needed for auditing/reporting.

status = LOCKED lockedBy = USER123 lockExpiresAt = now + 5 minutes

This blocks other users from redeeming the same coupon during the window and cuts race conditions on hot coupons.

4. Redeem within 5 minutes or release

Case A — User redeems in time: POST /coupons/redeem/{couponId}. Verify lock (Redis and/or DB), same user, not expired; then status = REDEEMED, write Redeem row, delete Redis lock key, update Elasticsearch.

Case B — User abandons or payment fails: If using Redis with TTL, the lock key expires after 5 minutes — no scheduler strictly required for Redis. Optionally a background job reconciles DB rows still marked LOCKED past lockExpiresAt to ACTIVE if DB and cache ever diverge.

status = ACTIVE lockedBy = null lockExpiresAt = null

Interview follow-ups: How is lock expiration implemented? Two users locking at once? DB row lock vs Redis? How is search kept consistent with DB?

Deep dive — Redis for locking (and optional ID generation)

Coupon locks are temporary (5 minutes), must auto-expire, see high contention on popular coupons, must be fast, and must avoid races. Redis fits: TTL, atomic SET NX, O(1) operations.

Consistency with search: After the DB transaction commits, notify an indexer (async) to update Elasticsearch — don’t treat ES as the transactional store.

Basic Redis lock

SET coupon:lock:C123 USER123 NX EX 300
  • NX — set only if the key does not exist (one winner).
  • EX 300 — expire after 300 seconds (5 minutes).
  • Return OK → lock acquired; null → someone else holds it.
  • No cron job required purely for Redis key expiry.

End-to-end with Redis

1) Lock — Coupon Redeem Service tries Redis; on success, optionally writes LOCKED to DB for visibility; Redis holds coupon:lock:<id> = userId with TTL.

2) Redeem — Check lock exists and user matches; apply business rules; insert Redeem row; set coupon REDEEMED; DEL coupon:lock:<id>; update ES.

3) Abandon checkout — TTL expires; key disappears; coupon becomes available again for others (reconcile DB if you mirror lock state there).

Unique coupon IDs (architecture diagram)

Coupon Generation Service can call Redis atomic INCR on a counter key, format the next sequence into couponId, then persist to the database — avoids collisions under concurrent admin uploads.

Deep dive — Elasticsearch for coupon search

At small scale, DB search can work. As data and read traffic grow (millions of coupons, 100:1 reads, free-text queries like “10% laptop discount”), SQL LIKE and heavy secondary indexes struggle: full scans, latency, expensive read replicas.

Rule: Database = source of truth. Elasticsearch = search-optimized projection only.

1) Index

E.g. coupons_index — a collection of documents optimized for query, not OLTP.

2) Document (example)

{ "couponId": "C123", "category": "electronics", "description": "10% off on laptops", "type": "PERCENTAGE", "status": "ACTIVE", "expiryDate": "2026-02-10" }

3) Fields

Fields are analyzed and indexed — e.g. description tokenized, category filterable, expiryDate for ranges.

4) Inverted index

Instead of only “row → text”, ES builds term → document IDs, e.g. laptop → [C123, C456, …]. Queries intersect/rank these lists quickly.

5) Shards

Split the index across shards on multiple nodes; queries run in parallel for horizontal scale.

6) Replicas

Per-shard copies for failover and extra read throughput for search.

7) What Elasticsearch is not for

  • Not a system of record for financial/locking state.
  • Weak at multi-row transactions, joins, strict immediate consistency, very chatty fine-grained updates.

Generation and redeem services update Elasticsearch after DB changes so “available coupon” search stays eventually consistent with the primary store.

Cross-cutting rules

  • One coupon per user per day — Enforce in application / API layer (e.g. check daily claim counter or ledger per user), in addition to per-coupon lock/redeem rules.
  • Single redemption — Enforced by DB uniqueness on couponId in redeem table or status transition guards, plus idempotent redeem API where appropriate.
  • CAP framing — Favor availability for search paths; favor correctness for allocate/lock/redeem paths (often backed by DB + Redis primitives).

Key insights

  • Redis TTL locks avoid relying only on slow polling or heavy DB row locks for the 5-minute window.
  • Redis INCR gives cheap, concurrent-safe unique IDs for high-volume coupon creation.
  • Elasticsearch delivers low-latency search with inverted indexes; index description, category, and type.
  • Async indexing after DB commit keeps search in sync without coupling write latency to ES.
  • Read-heavy (100:1) workloads benefit from separating OLTP (DB) from search (ES) and caching hot paths where needed.
