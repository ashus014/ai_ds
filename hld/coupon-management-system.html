Coupon Management System

Discount Coupon Generation & Validation — HLD Notes

Completed: 08 January 2026

Problem Statement

  • Design a coupon generation system
  • Allow users to create, manage, and validate discount coupons
  • One user gets a single coupon per day
  • User must buy within 5 minutes; otherwise coupon becomes available for others
  • Coupons cannot be redeemed multiple times

Functional Requirements

  • Admin can create 100k coupons daily
  • User can search coupons
  • User can get and redeem coupons
  • One user gets a single coupon per day
  • User must use coupon within 5 minutes
  • If not used in 5 min, coupon becomes available for others
  • System validates coupons (no multiple redemption)

Non-Functional Requirements

  • CAP: High availability for search/viewing; strong consistency for getting coupons
  • Latency: Low-latency search
  • Scale: 10M users for 100k coupons
  • Read heavy: 100:1 read/write ratio

Core Entities

  • Coupon — couponId, category, description, date, type, status
  • Redeem — redeemId, couponId, userId, orderId, timestamp
  • User — userId, firstName, lastName, email, phone

API Design

1. Create Coupon (Admin)

POST /coupons — Returns couponId. Body: category, description, expiryDate, type (PERCENTAGE/FLAT)

2. Search Coupon

GET /coupons/search?search_term=&type= — Returns list of coupons matching filters

3. Get (Lock) Coupon

POST /coupons/locked/{couponId} — Locks coupon for user. Status → LOCKED; lockedByUser, lockExpiresAt set. Returns coupon.

4. Redeem Coupon

POST /coupons/redeem/{couponId} — Body: userId, orderId. Validates, applies discount, marks REDEEMED.

Validations: Not expired, belongs to user (if personalized), not already redeemed, was locked (optional).

Object Modeling

Coupons

couponId, category, description, dateCreated, type (PERCENTAGE/FLAT), status (ACTIVE/LOCKED/REDEEMED/EXPIRED), createdTimeStamp, lockedTimeStamp, userId

Redeem

redeemId, couponId, userId, orderId, redeemTimeStamp

Users

userId, firstName, lastName, email, phoneNumber

High-Level Design Flow

  1. Admin creates coupon — POST /coupons → Coupon Generation Service → DB (status: ACTIVE)
  2. User searches — Search Service queries DB (basic) or Elasticsearch (optimized)
  3. User selects coupon — POST lock → status LOCKED, lockedBy, lockExpiresAt (5 min)
  4. Redeem or release — If redeemed within 5 min: REDEEMED. Else: background job resets to ACTIVE

Redis for Lock Mechanism

Redis fits coupon locking: temporary, auto-expire, high contention, fast, avoids race conditions.

Basic Redis lock:

SET coupon:lock:C123 USER123 NX EX 300

NX = set only if key does NOT exist; EX 300 = expire after 5 minutes. OK → lock acquired; null → already locked.

Flow with Redis

  • Lock: Try Redis lock → if success, proceed; if fail, reject. TTL 300s. No cron for expiry.
  • Redeem: Check Redis lock exists, user matches, apply discount, save Redeem record, mark REDEEMED, DEL lock
  • Abandon: Redis TTL expires automatically; coupon available again

Elasticsearch for Search

At scale: millions of coupons, high read traffic, free-text search. DB has full table scans, high latency.

Key concepts:

  • Index — Collection of documents (e.g., coupons_index)
  • Document — JSON per coupon (couponId, category, description, type, status, expiryDate)
  • Inverted Index — Word → list of documents (e.g., "laptop" → [C123, C456])
  • Shards — Index split for scale; parallel search
  • Replicas — Fault tolerance, read capacity

Rule: DB = source of truth; Elasticsearch = search only. DB not good for: transactions, joins, strict consistency, frequent updates.

Key Insights

  • Redis TTL for locking — no cron jobs; auto-expiry
  • Elasticsearch for low-latency search at scale
  • 5-minute lock prevents double allocation and race conditions
  • One coupon per user per day enforced at application layer
  • Strong consistency for get/lock; high availability for search
