Hotel Booking (Airbnb)

Accommodation Booking System — HLD Notes

Completed: 14 March 2026

Summary

Comprehensive design for a hotel booking system similar to Airbnb, MakeMyTrip, or Booking.com. Covers functional and non-functional requirements, API design, high-level architecture, database schemas, search optimization, and concurrency control for bookings.

Functional Requirements

  • Register and login
  • Search hotels by location, hotel name, and booking date
  • View hotel details and available rooms for a specific date
  • Book available rooms and complete payment
  • Access history of past bookings

Non-Functional Requirements

  • Scale: ~50 million users, ~1 million hotels globally
  • CAP Theorem:
    • High availability for search
    • High consistency for booking (prevent double booking)
  • Pagination for search results to reduce latency

Core Entities

  • User — Customer booking hotels
  • Hotel — Hotel entity
  • Room — Rooms within hotels
  • Booking — Booking record

High-Level System Architecture

  • Architecture: Microservices for scale
  • Load Balancer & API Gateway: Traffic, auth, rate limiting
  • User Service: Registration, authentication, metadata
  • Search Service: Hotel search via ElasticSearch
  • Booking Service: Booking requests, payment delegation
  • Payment Service: Third-party gateway integration
  • Booking Info Service: Past bookings
  • Review Service: Hotel reviews, separate metadata storage

Search Service and Optimization

  • ElasticSearch for:
    • Text search (hotel name)
    • Geospatial proximity search (location)
  • Alternatives considered: Quad Tree (large memory), Postgres GIS (limited text), GeoHash (no text search)
  • Two-phase search:
    • Phase 1: ElasticSearch queries hotel metadata (name, location)
    • Phase 2: Fetch room availability and pricing from Redis cache for filtered hotel IDs
  • Images from S3 included in search response

Booking Service and Payment Flow

Two-step booking:

  1. Booking request received; availability checked
  2. Payment via Payment Service and third-party gateway

Avoiding Double Booking

  • Availability verified via Availability Service before payment
  • Redis distributed locks with TTL (~5 min) to hold rooms during payment
  • Payment success → booking confirmed; payment fails or lock expires → room released
  • Kafka event-driven for: user notification, async booking DB update, availability update

Concurrency Control for Booking

ScenarioSolution
Two users book same room seconds apartReal-time availability check before payment
Nearly simultaneous booking attemptsRedis distributed lock (TTL) on room; only one booking proceeds

Key Insights

  • Requirement gathering is critical for core functionalities and constraints
  • Balance CAP by splitting availability (search) and consistency (booking) across services
  • ElasticSearch for combined text and location-based search
  • Event-driven (Kafka) for decoupled, reliable updates
  • Redis distributed locking for booking concurrency
  • Separate metadata and image storage for performance
