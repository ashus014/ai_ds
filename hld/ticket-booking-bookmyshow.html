Ticket Booking (BookMyShow)

Event Ticketing System — HLD Notes

Completed: 14 March 2026

Summary

Comprehensive design for a scalable ticket booking system similar to BookMyShow or PTM. Users search for events, view details, and book tickets for movies, concerts, or shows. Covers requirements, API design, high-level and low-level design, concurrency handling, and peak traffic management.

Functional Requirements

  • Search for events: By keywords, location, or date
  • View event details: Performers, venue, seat layout
  • Book tickets: Select seats and complete booking via payment

Non-Functional Requirements

  • Scale: Up to 100 million daily active users
  • CAP Theorem:
    • High availability for search and viewing
    • High consistency for booking to avoid double booking
  • Trade-off: Search prioritizes availability; booking prioritizes consistency

Core Entities

EntityDescription
UserEnd consumer booking tickets
EventMovies, concerts, or shows
VenueLocation or hall where event takes place
SeatIndividual seats available for booking

API Design

  • GET /search: Params — event title, location (lat/long), date; returns paginated event IDs and metadata
  • GET /event/{eventId}: Returns event metadata including seat availability
  • POST /booking/reserve: Input list of seat IDs; temporary reservation (5–10 min)
  • POST /booking/confirm: Input booking ID + payment details; finalizes booking

User ID passed in request headers for authentication.

High-Level System Design

  • Architecture: Microservices for scalability
  • Clients: Mobile/web users
  • API Gateway: Authentication, rate limiting, routing
  • Microservices:
    • Search Service — Search queries
    • Event Service — Event metadata retrieval
    • Booking Service — Seat reservations and bookings
  • Databases:
    • Event metadata → Cassandra (NoSQL, highly available, write-optimized)
    • Booking and payment → PostgreSQL/MySQL (strong consistency)

Low-Level Design — Data Storage

DatabaseData StoredStrengths
Cassandra (NoSQL)Event metadataHigh availability, write-optimized
PostgreSQL/MySQLBooking, payment infoStrong consistency
ElasticsearchSearch indexLow-latency, flexible search

Search Optimization

  • Cassandra write-optimized but not ideal for complex search
  • Elasticsearch for fast, flexible search
  • CDC pipeline syncs Cassandra → Elasticsearch in near real-time

CDC Pipeline Flow

  • Cassandra changes captured by Debezium
  • Events streamed via Kafka
  • Data ingested into Elasticsearch for instant search updates

Booking Service & Concurrency Handling

Two-step booking:

  1. Reserve seats — Temporary lock (5–10 min)
  2. Confirm booking — After successful payment

Avoiding Race Conditions (Double Booking)

  • Redis cluster with TTL as distributed lock for seat reservations
  • Seat reserved → entry in Redis with 10-minute expiry
  • Payment success → remove Redis lock, update persistent DB
  • Payment fails or timeout → Redis auto-releases lock (no cron job)
  • Eliminates unreliability of periodic cron jobs scanning for expired reservations

Handling Traffic Spikes

  • High-demand events (e.g., concerts) → waiting queue via Kafka
  • Incoming booking requests enqueued to manage load
  • Sequential dequeue for orderly processing; prevents system overload

Key Insights

  • Separation of concerns in microservices enhances scalability and maintainability
  • CAP trade-offs per use case: availability for search, consistency for booking
  • Redis TTL for seat locks provides clean concurrency control without cron jobs
  • Elasticsearch + CDC enables fast search over large event datasets
  • Kafka queue for peak traffic ensures stability during high demand
