Rate Limiting

Strategies and Algorithms — HLD Notes

Completed: 14 June 2025

Summary

Rate limiting is a crucial concept in system design used to control the number of requests a server processes to prevent overload and misuse. It covers the fundamental idea behind rate limiting, its importance, and various algorithms used to implement it.

Core Concepts

Rate Limiting Definition

Controls the number of requests a server can handle in a given time window to prevent server crashes or performance degradation.

Typical Scenario

Multiple users send requests to a server, which processes them and sends responses.

Server Limitations

Servers have hardware constraints (e.g., CPU, RAM) limiting how many requests can be processed per unit time. For example, a server might handle only 100 requests per minute safely.

Purpose of Rate Limiter

Introduce a layer that caps requests at a configured threshold (e.g., 100 requests/minute). Requests exceeding this limit are rejected with HTTP 429 Too Many Requests status code.

Rate Limiter Workflow

  • Checks if incoming requests exceed the threshold
  • Allows requests within limits to pass
  • Drops/rejects excess requests to protect server stability

Detailed Algorithm Insights

Token Bucket Algorithm

Mechanism: Tokens represent permission to process requests. Bucket has a capacity (e.g., 5 tokens) and refills tokens at a fixed rate (e.g., 3 tokens per second). Each incoming request requires a token; if available, request proceeds; if not, request is rejected (429 error).

Burst Handling: Allows short bursts if tokens are available, making it flexible and widely used in production (e.g., Amazon, Stripe APIs).

Challenges: Requires tuning bucket capacity and refill rate based on server capacity.

Leaky Bucket Algorithm

Mechanism: Requests queue up in a bucket. Requests are processed at a fixed rate regardless of incoming burst size. If bucket fills, excess requests are dropped.

Analogy: Like a shower controlling water flow from an overhead tank to prevent flooding.

Suitable Use Cases: When stable outflow rate is necessary.

Challenges: Tuning queue size and processing rate; risk of request starvation under heavy bursts.

Fixed Window Counter Algorithm

Mechanism: Time divided into fixed intervals (e.g., 1 second). Requests counted per window; if count exceeds threshold, further requests rejected until next window.

Issue: Burst problem where requests at window edges can exceed limits (e.g., 3 requests allowed per second, but 4 requests processed if some fall at end of one window and some at start of next).

Pros: Easy to implement and reset.

Sliding Window Log Algorithm

Mechanism: Maintains a timestamp log of each request. When a new request arrives, expired timestamps outside sliding window are removed. Requests are accepted or rejected based on counts within the sliding window.

Advantage: Addresses the burst problem of fixed windows by keeping precise request history.

Trade-off: More memory and computational overhead due to logging and cleaning timestamps.

Sliding Window Counter Algorithm

Mechanism: Combines fixed window counting with sliding window logic. Uses weighted counts of current and previous fixed windows to estimate request counts in a sliding manner. For example, if a request falls 10% into the current window, 90% weight is given to the previous window count and 10% to current.

Benefit: Reduces burstiness seen in fixed window counters while being more memory efficient than sliding window logs.

Use: Hybrid approach balancing accuracy and resource usage.

Key Takeaways

  • Rate limiting is essential for protecting servers from overload and ensuring fair resource allocation.
  • Different algorithms offer trade-offs between accuracy, complexity, memory usage, and burst handling.
  • Token Bucket and Leaky Bucket are common, practical algorithms with different focuses on burst and steady flow.
  • Fixed Window Counter is simple but suffers from burst problems at window edges.
  • Sliding Window Log and Sliding Window Counter improve accuracy by smoothing the counting over time, mitigating burst issues.
  • Proper tuning of parameters (bucket sizes, refill rates, queue length) is crucial for optimal performance.
  • HTTP 429 Too Many Requests is the standard error code for rejecting excess requests.
