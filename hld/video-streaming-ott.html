Video Streaming (Netflix/YouTube)

OTT Platform System Design — HLD Notes

Completed: 14 March 2026

Summary

Comprehensive system design guide for building an OTT (Over-The-Top) video streaming platform similar to Netflix, Amazon Prime, or Hotstar. Covers video processing, backend architecture, API design, and streaming protocols.

Functional Requirements

  • User registration and subscription management: Create accounts, subscribe (monthly/yearly) before accessing content
  • Content search: Search movies/shows by title or genre
  • Video playback: Multiple resolutions from 480p to 4K
  • Recommendation engine: Typically out of scope (complexity, AI/ML)

Non-Functional Requirements

  • Scale: ~200 million users globally, ~10,000 videos (~1 hour each)
  • CAP: Prioritize high availability for viewing; strong consistency for payment and subscription only
  • Latency: Zero or minimal buffering for seamless playback

Core Entities

  • User — Metadata including subscription status
  • Video — Actual video files in blob storage (e.g., S3)
  • Video metadata — Titles, descriptions, thumbnails, manifest files in metadata DB

High-Level System Architecture

  • Client: Interface for streaming and interaction
  • Load Balancer & API Gateway: Round-robin traffic; auth, routing, rate limiting
  • User Service: Registration, login, metadata; MySQL
  • Subscription Service: Plans, payments, status; Payment Gateway, Payment DB; Kafka for async updates
  • Search Service: Video metadata search via Elasticsearch
  • Play Service: Streaming; fetches manifest, adaptive streaming
  • Uploader Service: Backend uploads videos; not user-facing
  • Blob Storage (S3): Video files and thumbnails
  • CDN: Caches popular video chunks closer to users
  • Kafka: Event-driven; subscription payments, async updates

Video Processing Pipeline

  • Uploader Service uploads video and metadata
  • Blob Storage holds raw video and thumbnails
  • Chunker Service splits video into segments (2–10 seconds)
  • Kafka buffers chunked segments
  • Video Encoder Service encodes each chunk into multiple resolutions (480p, 720p, 1080p, 4K) using H.264/H.265
  • Encoded segments stored back in Blob Storage
  • Manifest file (playlist) maintains sequence and resolution variants:
    • M3U8 — HLS protocol
    • MPD — DASH protocol

Adaptive Streaming Logic

  • Client downloads manifest from Play Service
  • Client measures bandwidth and requests appropriate chunk resolution
  • Chunks fetched from CDN for lower latency
  • Quality adjusts dynamically based on real-time bandwidth
  • Each chunk starts with I-frame for smooth transitions

Key Takeaways

  • Separate concerns: user management, subscription, search, playback
  • High availability and low latency are critical for streaming
  • Pipeline: chunking → encoding → manifest files enables adaptive streaming
  • CDN reduces latency and buffering by caching content near users
  • DASH offers multi-bitrate prefetching; preferred by Netflix, Prime
  • Kafka for async subscription and notification updates
  • MongoDB for video metadata due to schema flexibility
