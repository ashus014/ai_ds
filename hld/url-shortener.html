URL Shortener (Bitly-like)

System Design — HLD Notes

Completed: 14 March 2026

Summary

Comprehensive system design walkthrough for building a URL shortening service, a common interview question. Introduces a structured framework: gathering requirements, defining core entities, designing APIs, and iterating from high-level to low-level design with scalability, reliability, and performance in mind.

Problem Statement

Design a system that converts long URLs into short, easy-to-share URLs. One-to-one mapping between each active short key and a long URL. Users provide a long URL; the system returns a shortened form. When the short URL is accessed, the user is redirected to the original long URL. Optional capabilities (custom short keys, expiration) can be layered on top of this core flow.

Functional Requirements

  • Create a short URL from a long URL
    • Optional: support a custom short URL (user-chosen slug)
    • Optional: support an expiration date (short link stops redirecting after that time)
  • Redirect — given a short URL, the user is sent to the original long URL

Non-Functional Requirements

  • Low latency: URL shortening and redirection within ~200 ms
  • High availability (redirect path): From a CAP perspective, the system should stay highly available for redirects—users should usually reach the redirect service and get a valid response even under partial outages or replication lag. We deliberately do not require the same bar for immediate consistency everywhere.
  • Scalability: ~100 million daily active users, ~1 billion URLs shortened
  • Uniqueness: Shortened URLs must be unique
  • Eventual consistency: It is acceptable if a newly created mapping is not reflected instantly on every replica or cache edge—there may be a short delay between creating a short URL and successfully hitting it. Availability over strict, immediate consistency is the right trade-off here: redirects should keep working; perfect read-your-writes globally is optional.

Low-Level Design — URL Shortening Approaches

1. Approch 1 : Using encryption Logic

  • There are few libraries freely availbale over internet for the encryption - MD5, SHA-1, base64
  • MD5 returns a 32-character encrypted string, and SHA1 returns a 40-character encoded string
  • But a shortened URL should be 5-6 characters long.
  • And what if first 6 charcters of two URL become same, then that might be the issue- So we need to check the database if the URL exists already or not
    • If not -> then return the same
    • If it exists -> append some key and generate once again
  • But with this approach -> we are generating url (1ms), then calling database(5ms) and then writing database(10ms)
  • If already found in database, then do it all over again - 16ms + 16ms

2. Counter-Based Approach

  • We are using a counter, and the encryption logic is in our hands—with this approach each and every request is unique (no collision from truncating a hash).
  • We save a lot of time not calling the database on every create to check if the key already exists.
  • Latency can drop compared to the hash-and-check path—for example from ~16ms down to ~6ms.
  • Issue: The main logic lives in the backend. If we scale the service, we end up with multiple pods, and each service can keep its own local copy of the counter. Then two instances can hand out the same value → duplicates.

Approach 2 (upgraded): counter in Redis

  • Instead of keeping a counter copy inside each microservice, we move the counter to Redis.
  • Accessing that value in Redis is very fast—about ~2–3ms.
  • Because Redis processes commands in a single-threaded way, counter updates are effectively atomic.
  • Issue: Redis is now a single point of failure. If Redis goes down, the entire architecture for issuing new IDs can fail.
  • We often need a Redis cluster for resilience—but then a similar problem can show up: each Redis instance can hold its own counter state. If one node fails and we read from another, counters may not line up → risk of duplicates unless we design the counter carefully for clustering (single logical counter, fencing, or pre-allocated ID ranges).

Approach 2 (continued): encryption vs decryption & Redis cache for reads

  • In the same server we are doing two things — (1) encryption and (2) decryption.
  • There could be a case where only about 20% of traffic is encryption (create short URL) and the rest ~80% is decryption (redirect / resolve).
  • So decryption should be scaled more than encryption—we should not over-spend resources on the encryption path.
  • With this we address separation of concerns (different scaling profiles for create vs resolve).
  • After all the optimisation we can still have latency on reads: to find the long URL we may hit the database again and again. We can put a Redis cache between the decryption servers and the database—request goes to cache first, then the database on a cache miss.
  • Then we do not need to hit the database on every lookup; roughly ~2ms Redis lookup is often enough.

3. Approach 3: Distributed unique IDs with ZooKeeper

  • Apache ZooKeeper is an open-source distributed coordination service for distributed applications.
  • It provides centralized building blocks for configuration management, naming, synchronization, and group services, so teams can manage large clusters with less ad-hoc glue code.
  • In practice ZooKeeper is used for things like keeping configuration in one place, naming / service discovery, distributed synchronization, distributed locking, and group membership (who is alive, who joined, who left).

ZooKeeper has two important znode types:

  1. Persistent node — as the name suggests, it is not tied to the client’s lifetime: it is not automatically removed from ZooKeeper when an application stops or disconnects. It stays in the namespace until an operator or process explicitly deletes that path.
  2. Ephemeral node — tied to a session between the ZooKeeper client (your app) and the ZooKeeper servers. The client must keep the session alive with periodic heartbeats (pings / keep-alives). If ZooKeeper decides the heartbeat has timed out—for example the application crashed or stopped sending traffic—it treats the session as dead and removes all ephemeral znodes that session created (they disappear from the tree under the relevant paths).

Approach 3: ZooKeeper — architecture walk-through

  • Counter in ZooKeeper (persistent layer). Any shared “counter” or sequence you keep in ZooKeeper for coordination lives under persistent znodes. You do not drive it like the per-request Redis counter from Approach 2—ZooKeeper is for registration / IDs / coordination, not the same hot-path increment pattern as Redis.
  • Worker ID on registration. When a server registers with ZooKeeper, ZooKeeper (or your registration recipe on top of it) gives that instance a unique worker ID.
  • One ID per instance. Three encryption server replicas → three worker IDs, for example 123, 234, and 345 (the diagram’s “two instances” idea generalizes: one worker ID per live instance).
  • Snowflake template (64 bits). To mint a unique numeric ID per request, use a Snowflake-shaped 64-bit value: the low 12 bits come from a local counter maintained inside each encryption server (sequence within the same millisecond / same worker), combined with sign, timestamp, and worker ID (see layout below).
  • On each create request, the microservice assembles sign bit + timestamp at request time + worker ID from ZooKeeper registration + local counter → a Snowflake ID unique across workers without calling ZooKeeper again.
  • What if ZooKeeper fails? ZooKeeper is used at server startup / registration for the worker ID—not on every shorten—so the steady-state create path does not depend on ZK being up for every request (new instances or re-registration still need a healthy ensemble).
  • Then reuse Approach 1 style encoding. Hash the 64-bit Snowflake (e.g. MD5 or SHA-1) and take a short prefix (for example the first six characters of the hex digest, mapped to your allowed alphabet) as the public slug. The token is derived from the Snowflake, not from re-hashing the whole long URL as the primary key source—so the chance of a accidental collision on that path is extremely low.
  • Snowflake ID generator (bit layout): 64-bit ID composed of:
    • 1-bit sign
    • 41-bit timestamp
    • 10-bit worker ID (from ZooKeeper)
    • 12-bit local counter
  • Benefits: IDs generated locally with uniqueness across servers (no per-create ZK or DB round-trip for the ID itself); very low collision risk after hashing; ZooKeeper touched on registration, not every shorten.

Additional Design Considerations

Separation of Concerns

  • Separate microservices for URL shortening (encryption) and redirection (decryption)
  • Scale decryption services more than encryption (redirection traffic ~80% of requests)

Caching

  • Redis cache for redirection lookups; reduces DB latency from ~10ms to ~2ms
  • Cache TTL matches URL expiration to avoid stale data

Expiration and Cleanup

  • Store expiration dates in DB and cache
  • Daily cron job to clean expired URLs from DB and cache

Custom URLs

When custom slugs are supported, check uniqueness in the DB (and reserved-word policy) before accepting a user-defined short key.

HTTP Redirect Status Codes

  • 301 (Permanent Redirect): Browser caches mapping; reduces server load but loses tracking
  • 302 (Temporary Redirect): Every request hits server; enables analytics and logging

Database Choice

  • Simple key-value store with indexing on short URL
  • PostgreSQL or MySQL sufficient
  • Index on short URL for efficient lookups

Key Insights

  • Requirement gathering and clarifying functional & non-functional needs are crucial first steps
  • CAP trade-off: high availability for redirects beats immediate global consistency—a brief lag after create before every edge sees the mapping is acceptable.
  • Effective scaling requires microservices, load balancing, caching, distributed ID generation
  • Zookeeper + Snowflake solves scaling and uniqueness; reduces latency and runtime complexity
  • Caching and separation of encryption/decryption services optimize resource utilization
  • HTTP status codes (301 vs 302) impact caching and analytics; choice depends on business needs
  • Always discuss trade-offs and limitations of each design choice in interviews

Conclusion

The design progresses from basic ideas (hashing) through practical improvements (counter with Redis) to a robust, scalable distributed system using Zookeeper and Snowflake IDs. The final solution balances performance, scalability, uniqueness, availability, and maintainability. Emphasizes structured problem-solving, API clarity, and system-level trade-offs in system design interviews.
