Ride Sharing (Uber/Lyft)

Ride Booking Application — HLD Notes

Completed: 15 March 2026

Summary

Comprehensive system design for a ride booking application similar to Uber or Ola. Covers functional and non-functional requirements, core entities, API design, high-level architecture, and low-level design for scalability, availability, and consistency.

Functional Requirements

  • Fare Estimation: Input start/drop locations → fare estimates for bike, car, SUV
  • Ride Request: User selects vehicle type and requests ride
  • Driver Matching: Match rider with nearby available driver
  • Driver Response: Drivers accept or decline based on fare and location
  • Real-Time Tracking: Rider and driver track each other's location
  • Rating System: Both parties rate each other post-trip
  • Payment Processing: Pay final fare through the app

Non-Functional Requirements

  • Scale: Millions of users and drivers
  • CAP Theorem:
    • High Availability (Users): Continuous ride request submissions
    • High Consistency (Drivers): Driver assignment consistent to prevent double allocation
  • Latency: Driver assignment within 1 minute; request fails otherwise

API Design Overview

User APIs

  • GET Fare Estimation: Input pickup/drop lat/long → list of fare estimates (vehicle type, request ID)
  • POST Ride Request: Input selected request ID → ride ID and driver info
  • GET Ride History: Past rides
  • POST Cancel Ride: Cancel by ride ID
  • POST Provide Rating: Submit rating post-ride

Driver APIs

  • WebSocket Location Update: Real-time driver location
  • POST Accept/Deny Ride: Request ID + acceptance status
  • POST Start Ride: Indicate trip start
  • POST End Ride: Indicate trip completion

High-Level System Architecture

  • Clients: Rider and driver mobile apps
  • Load Balancer & API Gateway: Auth, authorization, rate limiting; round-robin routing
  • Ride Service: Fare estimates via map APIs (Google/Apple Maps) for distance and ETA
  • Driver Matching Service: Geospatial proximity search; finds nearby drivers; sends notifications
  • Notification Service: Push to drivers via FCM (Android) / APNs (iOS)
  • Databases: Driver DB, Ride DB, Fare/Rate DB, Rating DB, Payment DB
  • Location Update Service: WebSocket server for real-time location
  • Redis Cache: Active driver locations for fast reads
  • ZooKeeper: Distributed locking to prevent double driver assignment

Low-Level Design — Key Technical Details

Fare Calculation

  • Third-party map APIs for route distance and ETA
  • Vehicle-specific rates and waiting charges from Fare DB
  • Surge pricing from real-time demand analytics

Driver Matching

  • GeoHashing and proximity search for drivers within radius (e.g., 5–10 km)
  • Query Redis cache for active drivers
  • Notify limited drivers (e.g., top 5) sequentially to meet 1-minute SLA
  • ZooKeeper distributed locks (ephemeral nodes) lock driver IDs during assignment; prevent race conditions
  • Locks released on driver response or timeout; auto cleanup on service crash

Real-Time Location Tracking

  • WebSocket connections via dedicated gateway; sticky sessions
  • Location updates every 5–10 seconds
  • Active driver locations cached in Redis; async processing to avoid DB overload
  • Kafka streams location updates for active rides; route tracking, fare reconciliation

Payment and Rating

  • Payment service integrates with external gateways
  • Payment data in relational DB with ACID properties
  • Rating service stores granular ratings; batch jobs aggregate to update driver/user averages

Key Insights

  • Separation of concerns: Microservices for fare, matching, notification, rating, payment
  • CAP balance: Availability for users; strong consistency for driver assignment
  • Proximity search: GeoHashing and Redis for fast nearby driver retrieval
  • Distributed locking: ZooKeeper for concurrency control and consistency
  • WebSocket: Low-latency real-time tracking for matching and tracking
  • Latency management: Limited driver notifications and timeouts for 1-minute SLA
  • Data strategy: Relational DBs for consistency; caches/queues for performance
