Food Delivery App (Zomato/Swiggy)

High-Level Design Notes

Completed: 18 March 2026

Actors

  • User – places order
  • Restaurant – accepts & prepares food
  • Delivery Agent – picks up & delivers food

Functional Requirements

  • Sign up / Login
  • Show nearby open restaurants (by location)
  • Search restaurant/menu by name
  • Show restaurant details & menu
  • Add to cart → Place order → Payment
  • Restaurant accepts order
  • Delivery partner gets allocated
  • Real-time delivery tracking for user

Non-Functional Requirements

  • Scale: 50M users, 1M restaurants
  • Architecture: Microservices (due to scale)
  • CAP Theorem:
    • Search/browsing → High Availability
    • Payments/ordering → High Consistency

Core Entities

User | Restaurant | Food/Menu Item | Delivery Agent | Payment

High-Level Design — Services

User Flow

User Service → User DB (MySQL/Postgres) — handles registration, login, JWT token generation

Search Flow

  • Search Service → Elasticsearch (for geo + text search) → queries Restaurant DB
  • Restaurant DB → Elasticsearch via CDC pipeline
  • Images stored in S3/Blob Storage (URLs saved in DB)

Cart Flow

  • Cart Service → Cart DB (Postgres)
  • One cart per restaurant (can't mix items from different restaurants)
  • Schema: cartId, restaurantId, userId, items[ ]

Order & Payment Flow

Order Service → calls Payment Service → 3rd party Payment Gateway → Payment DB (MySQL, highly consistent). On success, Order Service publishes event to Kafka

Kafka Event-Driven Flow (Order Lifecycle)

Order Placed
└── Kafka Topic: "order"
├── Order Consumer → updates Order DB
├── Notification Service → notifies User
└── Restaurant Acceptance Service → sends notification to Restaurant
└── Restaurant Accepts
└── Kafka Topic: "order_confirmed"
├── Order Consumer → updates Order DB (status: confirmed)
└── Delivery Matching Service → finds nearby driver
└── Kafka Topic: "driver_assigned"
└── Order Consumer → updates Order DB (adds driverId)

Delivery Matching — How It Works

  • Delivery Matching Service reads from Kafka (accepted orders)
  • Uses GeoHashing for proximity search to find nearby idle drivers
  • Sends notification to multiple nearby drivers via Delivery Acceptance Service → API Gateway → Drivers
  • First driver to accept gets assigned → confirmation event posted back to Kafka

Driver Location Updates

  • Drivers send location every 5–10 seconds
  • Goes via Kafka Gateway → Kafka Broker (handles massive load)
  • Location Update Service (consumer) → writes to Redis Cache (NOT directly to DB — too slow)
  • Delivery Matching Service reads from Redis Cache
  • TTL set on each driver entry (e.g., 10s) — if no update received, driver considered unavailable

Real-Time Tracking (User sees driver location)

  • WebSocket connection opened once order is picked up
  • WebSocket Manager — handles connections efficiently (memory-intensive)
  • Driver streams location → WebSocket Gateway → User

"Food Ready" Notification to Driver

Restaurant marks order as ready → event sent to Order Confirmation Service via Kafka. Order Confirmation Service sends push notification to assigned driver → driver goes for pickup

Schema Snapshots

  • User DB: userId, email, password, address, metadata
  • Restaurant DB: restaurantId, name, address, status (open/closed), lat/lng, imageURLs[ ]
  • Menu DB: menuId, itemId, restaurantId, title, description, imageURL, category, price
  • Cart DB: cartId, restaurantId, userId, items[ ]
  • Order DB: orderId, userId, restaurantId, items[ ], price, currency, timestamp, driverId
  • Payment DB: paymentId, userId, orderId, amount, currency, timestamp, status

Key Tech Choices & Justifications

ChoiceWhy
ElasticsearchGeo + text search together; fast for large-scale lookups
KafkaDecouples multi-step order lifecycle; prevents tight coupling between services
Redis Cache (for driver location)Thousands of drivers updating every 10s — DB writes too slow
WebSocketsReal-time bidirectional updates for live tracking
GeoHashingEfficient proximity search for nearby drivers
S3/Blob StorageStore restaurant/food images separately from metadata
MySQL for PaymentsStrong consistency required
Postgres for Orders/UsersRelational data with good query support
