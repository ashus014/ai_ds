Food Delivery App (Zomato/Swiggy)

High-Level Design Notes

Completed: 18 March 2026

Summary

Comprehensive system design walkthrough for a food delivery application similar to Zomato, Swiggy, or Uber Eats. Main goal: enable users to order food online from any restaurant. Covers functional and non-functional requirements, core entities, API design, high-level architecture, and low-level design including databases, services, event-driven architecture, and real-time updates.

Functional Requirements

  • User signup, login, logout, and profile update
  • Display all nearby open restaurants based on user location
  • Search restaurants by name or food menu items
  • View detailed restaurant metadata and menu items
  • Add multiple items to cart and manage cart (add, update, delete)
  • Place order with successful payment processing
  • Restaurant accepts the order and prepares food
  • Allocate delivery partner once order is accepted
  • Provide real-time tracking of delivery partner to the user

Actors

  • User — places orders
  • Restaurant — accepts and prepares orders
  • Delivery Partner — delivers food from restaurant to user

Non-Functional Requirements

  • Scale: ~50 million users globally, 1 million restaurants
  • Availability vs Consistency (CAP Theorem):
    • Search and browse → highly available
    • Payment and order placement → highly consistent
  • Architecture: Microservices with distributed systems
  • Support rate limiting, authentication, authorization, efficient load balancing

API Design Overview

User APIs

  • POST /register — signup with user metadata
  • GET /login, /logout
  • PUT /update — user details

Restaurant APIs

  • GET /restaurants?lat=&long=&radius= — list nearby open restaurants
  • GET /restaurants/search?title=&menu= — search by title and menu
  • GET /restaurant/{id} — restaurant details
  • GET /restaurant/{id}/menu — menu for restaurant

Cart APIs

  • POST /cart — add items (item ID, quantity, restaurant ID)
  • PUT /cart/{id} — update cart
  • DELETE /cart/{id} — remove cart

Order APIs

  • POST /order — place order (calls payment internally)
  • GET /order/{id}/tracking — track order and delivery partner

Internal APIs

Not exposed publicly: restaurant order acceptance, delivery partner allocation and status updates.

High-Level Architecture

  • API Gateway & Load Balancer: Routing, rate limiting, authentication, load distribution
  • User Service: Signup, login, profile; relational DB
  • Search Service: Location + text search via Elasticsearch
  • Restaurant Service: Restaurant metadata and menu updates
  • Cart Service: User carts with relational DB
  • Order Service: Order placement, payment coordination, order state tracking
  • Payment Gateway: Third-party integration
  • Delivery Matching Service: Assigns nearby idle delivery partners using geohashing
  • Location Update Service: Driver location updates; cache and database
  • Notification Service: Notifications to users and restaurants
  • Delivery Acceptance Service: Delivery partner confirmation
  • WebSocket Manager: Real-time delivery tracking

Search Implementation

  • Elasticsearch to index restaurant and menu metadata
  • Data updated via CDC (Change Data Capture) pipeline from restaurant DB to Elastic
  • Search queries use geolocation and text matching

Order Processing and Event-Driven Workflow

  • Order placement publishes event to Kafka
  • Consumers: update Order DB, notify user, notify restaurant for acceptance
  • Restaurant acceptance → event updates order status and triggers delivery partner assignment
  • Delivery Matching Service (Kafka consumer) assigns idle nearby drivers using geohashing
  • Delivery Acceptance Service notifies multiple drivers; first to accept is assigned
  • Delivery updates streamed via WebSocket for real-time user tracking

Delivery Partner Location Updates

  • Drivers send location every 5–10 seconds
  • Updates via Kafka gateway for high throughput
  • Location Update Service consumes events; stores in in-memory cache (e.g., Redis with TTL)
  • Cache used by Delivery Matching Service for efficient driver assignment

Key Insights and Conclusions

  • Microservice architecture essential due to scale (50M users, 1M restaurants)
  • Event-driven architecture (Kafka) ensures loose coupling and scalable order processing
  • Separation of concerns: distinct services for user, search, cart, order, restaurant, delivery, payment
  • Real-time tracking via WebSockets and efficient location update handling
  • Geohashing effective for delivery partner proximity search
  • Elasticsearch improves search for location and text-based queries
  • Strong consistency for payment/order; high availability for browsing/search
  • Multi-actor workflows with asynchronous notifications and state updates
  • Design can be progressively expanded (user → restaurant → delivery partner) depending on interview scope
