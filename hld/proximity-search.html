Proximity Search

Quad Tree vs Geohash vs PostGIS vs Elasticsearch — HLD Notes

Completed: 14 March 2026

Overview

Proximity search finds nearby locations (restaurants, hotels, drivers) within a radius from a user. Essential in Uber, Zomato, hotel booking systems. Four main approaches covered: Quad Tree, Geohashing, PostgreSQL GIS, Elasticsearch.

1. Quad Tree

Definition

Tree data structure where each node has four children, recursively partitioning 2D geographic space into four quadrants.

Implementation

  • Entire world divided into four quadrants
  • Each quadrant subdivided recursively until fine granularity (one or few data points)
  • Leaf nodes represent actual data points or small clusters

Pros

  • Simple to explain in interviews

Cons

  • Assumes uniform data distribution — unrealistic; real data is uneven
  • Unbalanced trees: dense areas have many subdivisions, increasing height and search complexity
  • Search requires traversing multiple nodes and parent-child backtracking — time-consuming
  • Data structure, not a database — significant memory and auxiliary space
  • Updating with new points is costly; partial or full rebuilds (like AVL/BST)
  • Concurrency: separate read/write copies add overhead

Conclusion: Not recommended for real-world production due to complexity, maintenance challenges, and inefficiency.

2. Geohashing

Definition

Spatial indexing that encodes lat/long into a string via recursive quadrant subdivision and base64 encoding.

Implementation

  • Divide world into quadrants labeled with numbers/characters
  • Subdivide recursively; each step encoded as a character forming the geohash string
  • Nearby locations share similar geohash prefixes; last character change = adjacent areas

Advantages

  • Easy to compute and compare geohash strings for proximity
  • Efficient neighbor lookup by manipulating a few characters
  • Precision levels control radius: ~5 km (level 5), ~500 m (level 6)
  • Well-suited for fast location-only proximity search
  • Simple to store geohash with lat/long in a database

Conclusion: Powerful and practical for pure location-based proximity search (e.g., nearby drivers).

3. PostgreSQL with GIS Extension (PostGIS)

Description

PostgreSQL with GIS capabilities — built-in spatial types and functions.

Features

  • Direct storage of geographic points (lat/long)
  • Spatial functions (e.g., ST_DWithin) for radius queries
  • Combine spatial queries with indexing and other DB features

Advantages

  • Mature, well-supported
  • Complex queries: proximity + other filters
  • Extensible with fuzzy string matching for text search

Use case: Location + additional attributes or text (e.g., restaurants by name and proximity).

4. Elasticsearch with Geolocation

Description

Elasticsearch natively supports geolocation queries and indexing.

Features

  • Built-in geospatial data types
  • Fast, scalable proximity search
  • Out-of-the-box full-text and fuzzy search
  • No extra extension installation

Advantages

  • Easy integration
  • Combined geo + text search naturally

Use case: Both textual and proximity search at scale (e.g., restaurants by name and location).

Key Insights

  • Avoid Quad Trees in production — complexity and inefficiency
  • Geohashing for pure location-based proximity search
  • For spatial + textual search: PostgreSQL GIS or Elasticsearch
  • Geohash precision: tune radius from km to meters
  • Store geohash with lat/long for efficient prefix-matching queries
