Building Real-Time Applications

Concepts and Techniques — WebSocket, Polling, Server-Sent Events

Completed: 20 March 2026

Summary

An in-depth look at how to build real-time applications: communication between servers and clients, and techniques to stream and update data in real time. The discussion uses practical examples such as a stock market application to surface challenges and ways to keep client data current.

Core Concepts and Problem Statement

  • Real-time application: The server continuously updates clients with fresh data as changes happen.
  • Traditional request-response model: The client sends a request; the server processes it (database, API, computation), responds, and the connection closes. After that, the server cannot push new data to the client, so the client’s view can become stale when backend data changes.
  • Example: In a stock app, after a one-off response, price moves on the exchange are not reflected until the client asks again.

Real-Time Communication Techniques

Technique Description Pros Cons Use case examples
WebSocket Stateful, full-duplex connection; both sides can send anytime. Real-time bidirectional updates; low latency; one continuous connection. Harder to scale horizontally; connection state on servers; complex load balancing. Chat, collaborative apps.
Polling Client asks for updates on a schedule. Simple, stateless; scales with load balancers. Request overhead; latency tied to interval. Stock tickers, periodic updates.
• Short polling Frequent requests; server responds immediately (no hold). Simple; server frees resources quickly. Many requests; wasteful if data rarely changes. High-frequency refresh needs.
• Long polling Server holds the request until new data (or timeout), then responds. Fewer round trips than short polling; near real-time. Many open connections; can be resource-heavy. Notifications, chat-style updates.
Server-Sent Events (SSE) Server pushes over one long-lived HTTP connection (server → client). Simpler than WebSocket for one-way push; continuous updates. Not for client → server on that stream (unidirectional). Live feeds, notifications.

Detailed Insights

WebSockets

  • Full duplex: Open channel; server can push anytime; client can send anytime.
  • Established via an upgrade handshake.
  • Scalability: Stateful connections are harder to spread across many servers; load balancers must be connection-aware.

Polling

  • Stateless pattern: client repeatedly asks for updates; works well behind load balancers.
  • Short polling: Fixed interval (e.g. every 2s); immediate response, with or without new data.
  • Long polling: Hold until data or timeout; client immediately opens the next request.
  • Long polling cuts request count vs short polling but still ties up connections; large fleets often favor polling for simpler scaling and fault tolerance.

Server-Sent Events (SSE)

  • Server streams updates on a single HTTP connection.
  • Unidirectional: server → client only on that channel (REST or other paths for client → server).
  • Good when you only need live server-driven updates.

Trade-offs and Choosing an Approach

  • WebSocket when you need true two-way streaming (chat, collaboration).
  • Polling (short or long) when simplicity and horizontal scale matter and the client mostly pulls data (e.g. markets with occasional REST actions).
  • SSE as a middle ground: server push without duplex on one connection.
  • Weigh: change frequency, client count, infra and ops budget, and team complexity.
  • Example: a stock app where users mostly read prices and send occasional buy/sell via REST may suit polling better than WebSockets.

Key Insights

  • Real-time UX needs more than one-shot HTTP request-response.
  • WebSockets excel at two-way, low-latency streams but are harder to scale out.
  • Polling trades some latency and load for simpler, stateless scaling.
  • SSE covers server-to-client streaming with moderate complexity.
  • Match direction of data flow, scale, and freshness to the mechanism you pick.

Conclusion

The material walks through why stale data happens after plain HTTP, then compares WebSockets, polling (short and long), and Server-Sent Events with pros, cons, and typical use cases. Pick based on product needs, scale, and infrastructure rather than a single “best” technology.
