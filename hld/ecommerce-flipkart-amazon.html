E-commerce Platform — HLD Notes

Completed: 14 March 2026

Summary

Comprehensive system design for a scalable e-commerce platform similar to Amazon or Flipkart. Covers requirement gathering, high-level architecture, API design, and low-level design including database choices, search optimization, caching, and concurrency control for limited stock.

Functional Requirements

  • User login/signup and authentication
  • Product search by keywords or title
  • View product details (description, images, manufacturer, reviews)
  • Add products with quantities to cart
  • Checkout and payment processing
  • Order status tracking
  • Handle concurrent purchases for limited stock (e.g., flash sales)

Non-Functional Requirements

  • Scale: 10 million monthly active users, 10 orders/second
  • Latency: Search response ~200 ms
  • CAP Theorem:
    • Search: High availability (users must always search)
    • Payment, orders, inventory: Strong consistency
  • Microservices architecture, round-robin load balancing
  • API Gateway: routing, auth, rate limiting

Low-Level Design Highlights

  • User Service: MySQL for structured user data
  • Product Service: MongoDB for flexible metadata; images in S3/CDN
  • Search Service: ElasticSearch synced via CDC (Kafka) from Product DB
  • Cart Service: PostgreSQL (cart_id, user_id, product_ids, quantities); price fetched from Product Service for freshness
  • Checkout flow:
    • Validate inventory via Inventory Service before payment
    • Payment via Payment Service + third-party gateway
    • Kafka events post-payment: update Order DB, Inventory DB, Notification Service
  • Inventory Service:
    • Source-of-truth in PostgreSQL
    • Redis cache to reduce DB load; write-through or CDC for consistency
    • Redis distributed lock with TTL for limited-stock concurrent purchases

Concurrency and Race Condition Handling

  • Multiple checkouts can query inventory for limited-stock items simultaneously
  • Redis distributed lock ensures only one process decrements inventory per product
  • Lock TTL (e.g., 30 sec–1 min) to avoid deadlocks
  • If lock held by another process → request fails gracefully; user notified of unavailability

Key Insights

  • Balance consistency and availability per CAP theorem
  • Microservices with clear separation enhance scalability and maintainability
  • Event-driven (Kafka) decouples services; improves reliability in payment/order flows
  • ElasticSearch and Redis critical for search and caching at scale
  • Concurrency handling for limited stock prevents overselling and builds trust
  • Design addresses latency, scalability, and fault tolerance
