Social Media Platform (Facebook/Instagram)

System Design — HLD Notes

Completed: 15 March 2026

Summary

Comprehensive approach to designing a large-scale social media platform like Facebook or Instagram for system design interviews. Covers functional and non-functional requirements, core entities, API design, high-level architecture, and low-level design with optimization techniques.

Key Functional Requirements

  • User onboarding: Sign up and login
  • Post creation and sharing: Text, image, and video posts
  • Follow system: Users can follow each other (simplified; no friend requests)
  • User interactions: Like and comment on posts
  • User feed: Display feed from followed users, prioritized by relevance

Out of scope: Stories, messaging/chat, other advanced features.

Non-Functional Requirements

  • Scale: ~500 million daily active users
  • Availability vs Consistency: High availability and eventual consistency over strong consistency
  • Latency: Average upload latency ~500 ms for posts (text/image)

Core Entities

  • User — Individual using the platform
  • Post — Content (text, images, videos)
  • Follower/Friend relationship — Who follows whom
  • Engagement — Likes and comments on posts
  • Feed — Aggregated posts visible on login

Pagination used for feeds and user posts to handle large volumes.

High-Level System Architecture

  • Clients: Mobile apps or web browsers
  • Load Balancer & API Gateway: Auth, authorization, rate limiting; round-robin routing
  • Microservices:
    • User Service — Onboarding, metadata; Postgres
    • Content Service — Post creation; text in Cassandra, media in blob storage (S3)
    • Follower Service — Follower relationships; Postgres
    • Engagement Service — Likes and comments; Kafka + Cassandra
    • Feed Service — Generates and serves feeds from feed DB and cache
    • Moderator Service — AI/ML content filtering before publishing
    • Notification Service — Alerts users on blocked posts
    • Fan-out Service — Propagates new posts to followers' feeds
    • Backfill Service — Generates feeds in near real-time for infinite scroll

Deep Dive: Design Details & Optimizations

Content Upload Flow

  • Posts buffered in Kafka topic ("raw post")
  • Moderator service consumes raw posts, filters, publishes to "filtered post" or "blocked post" topics
  • Post consumer service ingests filtered posts → Cassandra (metadata) + blob storage (media)
  • Blocked posts trigger notifications to users

Cassandra for write-heavy, low-latency posts and comments; Postgres for users and followers.

Feed Generation: Challenges and Solutions

  • Naïve approach (join followers + posts at runtime) infeasible at scale
  • Fan-out design with two models:
    • Push model: Normal users with fewer followers → update feeds immediately on post creation
    • Pull model: Celebrities with millions of followers → fetch posts on-demand when feed requested
  • Feed precomputed and stored in feed DB + cache
  • Feed cache (e.g., Redis) in front of feed DB for faster reads
  • Fan-out service creates feed entries after moderation; publishes to Kafka
  • Fan-out consumer consumes Kafka, updates feed DB and cache
  • Feed size limits (e.g., 50–100 posts per user) with TTL
  • When cached feed exhausted, backfill service generates more posts in near real-time
  • Post materializer maintains recent user activity in Redis for faster feed generation
  • Top follower cache — stores frequently interacted followers to optimize relevance

Key Insights & Conclusions

  • Microservices and distributed architecture essential for hundreds of millions of users
  • Prioritize availability and eventual consistency for social media UX
  • Message brokers (Kafka) enable buffering, async processing, resilience in write-heavy scenarios
  • Caching layers (feed cache, Redis for activity and top followers) critical for latency and DB load
  • Fan-out models balance real-time updates with scale: push for average users, pull for high-follower users
  • Trade-offs between complexity and performance, especially in feed generation and moderation
  • API design maps closely to functional requirements
  • Postgres for structured user/relationship data; NoSQL for write-heavy post and engagement data
