Notification System

Scalable Design — HLD Notes

Completed: 10 December 2025

Summary

Detailed walkthrough of designing a scalable notification system for modern applications (e.g., social media). Focuses on engineering challenges, architectural improvements, and design patterns to handle emails, in-app notifications, and push notifications efficiently.

Core Concepts and Initial Design

  • A notification system triggers messages (emails, in-app, push) when user actions occur (sign-up, posts, friend requests)
  • Basic example: On sign-up, backend creates DB entry and sends welcome email synchronously via external API (e.g., Gmail API)
  • Problems with synchronous design:
    • Added latency: External API calls ~3 seconds → delays sign-up response
    • System dependency: Email provider down → entire sign-up flow crashes
    • Rate limiting: Providers limit (e.g., 30 emails/sec); high traffic → dropped requests

Improving Architecture: Asynchronous Processing

  • Transition to asynchronous email sending via separate internal email server/worker
  • Sign-up sends request to internal server; internal server calls external email API
  • Benefits:
    • Sign-up completes without waiting for email
    • Loosely coupled from external service failures
    • Email server failures don't directly affect sign-up flow
  • Remaining issue: External rate limiting still causes failures if too many requests hit email server

Queue-Based Architecture and Rate Limiting

  • Introduce queue (FIFO) to buffer email requests
  • Worker continuously polls queue to send emails
  • Rate limiting in worker: Send in batches (e.g., 25/sec to stay under 30); pause (e.g., 2s) when limit reached
  • Retry with exponential backoff: Failed sends retry after 1s, 2s, 4s, etc.; max retries (e.g., 10); messages exceeding limit → Dead Letter Queue (DLQ) for manual inspection
  • Drastically improves robustness and scalability

Supporting Multiple Notification Channels

  • Initially: single email queue
  • Evolving requirements: login notifications, friend requests, posts → multiple channels:
    • Email
    • In-app notifications
    • Push (e.g., Firebase)
  • Naive approach: separate queues per channel → high coupling, scattered code changes

Event-Driven and Fan-Out Architecture

  • Use event-driven architecture with pub/sub (e.g., AWS SNS)
  • Publish events: user.sign_up, user.login, friend.request
  • Multiple queues (email, in-app, push) subscribe to relevant topics
  • Fan-out: One event distributed to multiple channels without changing core logic
  • Benefits: Minimal code changes for new channels; flexible binding; easier maintenance and scale
  • Workers consume from queues and send notifications

Intelligent Notification Triggering and User Preferences

  • Intelligent filtering: Not all events need all notification types for every user
  • Skip email if user is online or already viewed content
  • Respect user preferences (enable/disable email, push, in-app)
  • Business rules:
    • Don't email every chat message; batch and summarize
    • Suppress in muted channels or "Do Not Disturb"
    • Notify only when mentioned or subscribed
  • Ingestion/filtering layer reduces noise and improves UX

Important Design Patterns and Techniques

  • Asynchronous processing: Reduce latency, decouple external dependencies
  • Queue-based buffering: Handle traffic spikes and rate limits
  • Retry with exponential backoff and DLQ: Avoid infinite loops; enable manual recovery
  • Pub/Sub fan-out: Decouple producers and consumers; simplify multi-channel dispatch
  • Ingestion/filtering layer: Control which notifications are sent based on context and preferences

Key Insights

  • Notification systems are complex; must handle latency, failures, rate limiting, and UX
  • Asynchronous queues and event-driven design improve scalability and fault tolerance
  • Fan-out with pub/sub enables flexible, maintainable multi-channel notifications
  • Filtering and user preference management are critical to avoid spamming
  • Solutions like Courier, SNS+SQS can accelerate development for startups

Conclusion

Designing a robust notification system involves:

  • Avoiding blocking user flows with synchronous external calls
  • Handling API failures and rate limiting via queues and retries
  • Supporting multiple channels with minimal coupling using event-driven fan-out
  • Incorporating smart filtering based on user context and preferences

This layered, scalable architecture ensures reliable, user-friendly notification delivery at scale.
