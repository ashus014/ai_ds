Centralized Logging (Splunk)

Distributed Logging System — HLD Notes

Completed: 18 March 2026

Summary

System design for a distributed logging platform that collects, processes, and exposes logs from multiple microservices globally. Focus on scalability, reliability, and near real-time availability — similar to Splunk.

Functional Requirements

  • Multi-source ingestion: Logs from microservices deployed worldwide
  • Near real-time submission: Low latency from generation to availability
  • Offline/batch submission: Support reconnecting clients uploading logs
  • Validation and normalization: Parse and convert to standard format
  • Dashboard and search UI: Search for developers and support engineers

Non-Functional Requirements

  • Scale: Millions of log events per hour
  • Low latency: Minimal delay from generation to availability
  • CAP: Availability over consistency
  • Reliability: No data loss during ingestion
  • Retention: Configurable TTL (e.g., 30–45 days)

Core Entities

EntityDescription
Log EventIndividual log entries from microservices
Injection JobBatch job for offline log submission
Agent/SourceClient agent (e.g., Fluent Bit) that collects and forwards logs

High-Level System Design

  • Clients/Microservices: Log4j → stdout/stderr
  • Agent: Reads container log files; forwards to backend via API Gateway
  • API Gateway & Load Balancer: Auth, rate limiting, routing
  • Client Onboarding Service: Registers organizations; issues client ID and token
  • Injection Service: Real-time (agent) and batch (file) ingestion
  • Kafka Broker: Buffer and queue for high ingestion; decoupled processing
  • Apache Flink: Validation, parsing, normalization, enrichment, de-duplication
  • Log DB (Cassandra): Write-optimized storage for validated logs
  • Elasticsearch: Fast text search for recent logs (e.g., last 14 days)
  • Cold Storage (S3): Long-term archive for older logs
  • Search Service: Queries Elasticsearch (recent) and Cassandra (older)
  • Dashboard/UI: WebSocket streaming or polling
  • Alerting & Notification: Error detection, alerts, email/SMS/pager

Log Flow

Microservice → Log4j (stdout/stderr) → Kubernetes captures → Agent (Fluent Bit) → API Gateway → Injection Service → Kafka → Apache Flink → Cassandra & Elasticsearch → Search Service → Dashboard

Client Onboarding

  • Organizations register once; receive client ID and token
  • Postgres schema: client_id, token, token_ttl, environment (dev/staging/prod)

Kafka Topics

  • Raw logs: Unprocessed logs
  • Validated/enriched logs: Processed logs
  • Alert topic: Error logs that trigger alerts

Apache Flink Responsibilities

  • Validation (malicious content, incomplete logs)
  • Parsing and normalization (standard formats)
  • Enrichment (metadata)
  • De-duplication
  • Write to Cassandra, Elasticsearch, S3

Storage Tiers and Retention

StorageRetentionPurposeTradeoff
ElasticsearchLast 14 daysFast text search for dashboardCostly but performant
Cassandra30–45 daysPersistent log storageWrite-optimized, scalable
S3 / BlobLong-termCold storage, complianceCost-effective, slower

Soft delete from Cassandra after TTL; cold storage retains for compliance.

Search Optimization

Index by log level, trace ID, service name for quick filtering.

Alerting Design

  • Flink identifies error logs → pushes to Kafka alert topic
  • Alert Service consumes; evaluates rules (Alert DB); triggers Notification Service
  • Preferences (email, SMS, pager) stored as client metadata

Dashboard Log Streaming

  • WebSocket: Continuous near real-time streaming
  • Polling: Periodic fetch (every few seconds)

Key Insights

  • Separation of concerns: ingestion, processing, storage, search, alerting
  • Kafka + Apache Flink for scalable stream processing
  • Multi-tier storage balances cost and performance
  • Availability over consistency for log ingestion
  • Organization-level onboarding simplifies auth
  • Alerting adds proactive monitoring
