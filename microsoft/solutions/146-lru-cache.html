146. LRU Cache

Least Recently Used cache with fixed capacity: evict the least recently used entry when full. ⚠️ Naive scans a list; ℹ️ Map-assisted adds HashMap key → list index; ✅ O(1) LRU uses HashMap key → doubly-linked list node.

Microsoft · Medium

Summary

Store key–value pairs with cap n. get / put promote the entry to most-recent. Start from a plain list (scan every time), add a map from key → index (skip the scan, still pay for list shifts), then use a map from key → DLL node for true O(1) splice and eviction.

Answers

⚠️ Naive approach

List of [key, value] pairs; MRU at index 0. get scans, then move-to-front. put scans for an existing key or inserts at front; evict tail when at capacity.

  • Linear scan on every get / put.
  • No auxiliary structure — easy to explain, slow on large capacity.
⚠️ LRUCache.java — naive 
import java.util.*;

class LRUCache {

    List<int[]> cache;
    int n;

    public LRUCache(int capacity) {
        this.n = capacity;
        this.cache = new ArrayList<>();
    }

    public int get(int key) {

        for (int i = 0; i < cache.size(); i++) {
            if (cache.get(i)[0] == key) {

                int value = cache.get(i)[1];

                // move to front (most recent)
                cache.remove(i);
                cache.add(0, new int[]{key, value});

                return value;
            }
        }

        return -1;
    }

    public void put(int key, int value) {

        // check if key already exists
        for (int i = 0; i < cache.size(); i++) {
            if (cache.get(i)[0] == key) {

                // remove old position
                cache.remove(i);

                // insert at front (most recent)
                cache.add(0, new int[]{key, value});
                return;
            }
        }

        // if capacity full → remove least recently used (last element)
        if (cache.size() == n) {
            cache.remove(cache.size() - 1);
        }

        // insert new at front
        cache.add(0, new int[]{key, value});
    }
}
⚠️ Naive. Every get / put scans the whole list for the key — O(n) time plus list remove/insert. Fine for learning; not for production-scale caches.

ℹ️ Optimised (map + list)

Map<Integer, Integer> stores key → index in the list. After remove / add(0, …), call updateMap() to refresh every index. On eviction, remove the tail pair and map.remove(removedKey).

  • O(1) key lookup in the map; list shifts and updateMap() remain O(n) overall.
  • Good stepping-stone before a full DLL + hash O(1) design.
ℹ️ LRUCache.java — map + list 
import java.util.*;

class LRUCache {

    List<int[]> cache;
    Map<Integer, Integer> map; // key → index
    int n;

    public LRUCache(int capacity) {
        this.n = capacity;
        this.cache = new ArrayList<>();
        this.map = new HashMap<>();
    }

    public int get(int key) {

        if (!map.containsKey(key)) return -1;

        int index = map.get(key);
        int value = cache.get(index)[1];

        cache.remove(index);
        cache.add(0, new int[]{key, value});

        updateMap();

        return value;
    }

    public void put(int key, int value) {

        if (map.containsKey(key)) {
            int index = map.get(key);
            cache.remove(index);
        }
        else if (cache.size() == n) {
            int removedKey = cache.get(cache.size() - 1)[0];
            cache.remove(cache.size() - 1);
            map.remove(removedKey);
        }

        cache.add(0, new int[]{key, value});

        updateMap();
    }

    private void updateMap() {
        for (int i = 0; i < cache.size(); i++) {
            map.put(cache.get(i)[0], i);
        }
    }
}
ℹ️ Map-assisted. You skip the linear search for the key, but remove(index), insert at 0, and updateMap() still scale linearly with capacity — same asymptotic class as naive, with a better leading constant on lookup.

✅ Optimised (HashMap + doubly linked list)

Map<Integer, Node> maps each key to its list node. Sentinel head / tail dummy nodes bound the list so remove / insert never touch null. After get or put, move the node next to head (MRU); evict tail.prev when map.size() > capacity.

  • get / put: O(1) expected — hash lookup plus constant-time splices.
  • Space: O(capacity) for the map and one node per cached entry.
✅ LRUCache.java — HashMap + DLL 
import java.util.*;

class LRUCache {

    class Node {
        int key, value;
        Node prev, next;

        Node(int k, int v) {
            key = k;
            value = v;
        }
    }

    Map<Integer, Node> map;
    int capacity;

    Node head, tail;

    public LRUCache(int capacity) {
        this.capacity = capacity;
        map = new HashMap<>();

        // dummy nodes (avoid null checks)
        head = new Node(0, 0);
        tail = new Node(0, 0);

        head.next = tail;
        tail.prev = head;
    }

    public int get(int key) {

        if (!map.containsKey(key)) return -1;

        Node node = map.get(key);

        // 🔥 move to front
        remove(node);
        insert(node);

        return node.value;
    }

    public void put(int key, int value) {

        if (map.containsKey(key)) {
            remove(map.get(key)); // remove old
        }

        Node node = new Node(key, value);
        insert(node);
        map.put(key, node);

        // 🔥 remove LRU
        if (map.size() > capacity) {
            Node lru = tail.prev;
            remove(lru);
            map.remove(lru.key);
        }
    }

    private void remove(Node node) {
        node.prev.next = node.next;
        node.next.prev = node.prev;
    }

    private void insert(Node node) {
        node.next = head.next;
        node.prev = head;

        head.next.prev = node;
        head.next = node;
    }
}
✅ Production-style LRU. Hash map for key → node plus doubly linked list for recency order gives O(1) get and put without array shifts or full-map reindexing.

Complexity

⚠️ Naive

  • get(key), put(key, value): O(n) for scan + list ops.
  • Space: O(capacity).

ℹ️ Map-assisted

  • Map lookup O(1); list + updateMap() dominate — still O(n) per operation worst case.
  • Space: O(capacity) for list + map.

✅ HashMap + DLL

  • get(key), put(key, value): O(1) average — map lookup and constant pointer updates.
  • Space: O(capacity) for map plus one Node per entry (plus two sentinels).
