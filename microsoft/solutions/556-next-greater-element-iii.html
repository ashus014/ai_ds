556. Next Greater Element III

Given a positive integer n, find the smallest integer strictly greater than n that uses the same multiset of digits (leading zeros are not allowed). If none exists or the result exceeds 2³¹ − 1, return -1. Naive: enumerate permutations and track the best answer in a long. Optimal: one next permutation on the digit string in O(d) time.

Microsoft · Medium — 14 April 2026

Summary

Hint (naive): Generate all rearrangements of the digits of n via backtracking; at each leaf, parse the string as a long and, if it is > n and ≤ Integer.MAX_VALUE, update a running minimum with Math.min. Sentinel Long.MAX_VALUE means “no valid candidate yet.”

Hint (optimal): This is exactly next permutation on the digit array: find the rightmost ascent a[i] < a[i+1], swap a[i] with the smallest digit to its right that is larger than a[i], then reverse the suffix after i so the tail is as small as possible. One pass in O(d) time for d digits.

Answers

Naive approach — enumerate permutations, keep best with a long

Same factorial exploration as “generate everything then sort,” but avoids storing all permutations: whenever a full arrangement parses to a value > n and ≤ Integer.MAX_VALUE, fold it into result with Math.min. Using long avoids overflow while comparing against Integer.MAX_VALUE. Still not competitive with next permutation for large d.

Solution.java — permutations + min long 
import java.util.*;

class Solution {
    long result = Long.MAX_VALUE; // FIX: use long

    public int nextGreaterElement(int n) {
        char[] arr = String.valueOf(n).toCharArray();

        generate(arr, 0, n);

        // if still not updated → no answer
        return result == Long.MAX_VALUE ? -1 : (int) result;
    }

    private void generate(char[] arr, int index, int n) {
        if (index == arr.length) {
            long num = Long.parseLong(new String(arr));

            if (num > n && num <= Integer.MAX_VALUE) {
                result = Math.min(result, num);
            }
            return;
        }

        for (int i = index; i < arr.length; i++) {
            swap(arr, index, i);
            generate(arr, index + 1, n);
            swap(arr, index, i); // backtrack
        }
    }

    private void swap(char[] arr, int i, int j) {
        char temp = arr[i];
        arr[i] = arr[j];
        arr[j] = temp;
    }
}
Why this is still slow. You still visit on the order of d! leaf permutations (duplicate digit strings get explored many times). There is no sort pass anymore, so you save memory versus collecting every permutation in a list — but time remains factorial. The ✅ next-permutation solution below finds the answer in one linear pass over the digits.

✅ Optimal approach — next permutation (linear in digit count)

✅ Solution.java — next permutation 
class Solution {
    public int nextGreaterElement(int n) {
        // Convert number to char array so we can manipulate digits easily
        char[] arr = String.valueOf(n).toCharArray();

        // ---------------- STEP 1 ----------------
        // Find the "break point" (first dip from right)
        // We are looking for a place where:
        // arr[i] < arr[i + 1]
        // Why? Because that's where we can increase the number

        int i = arr.length - 2;
        while (i >= 0 && arr[i] >= arr[i + 1]) {
            i--; // keep moving left while it's non-increasing
        }

        // If no such index found → digits are in descending order
        // Example: 54321 → already the largest permutation
        if (i < 0) return -1;

        // ---------------- STEP 2 ----------------
        // Now we want to increase arr[i] slightly
        // Find the smallest digit on right side which is > arr[i]
        // We scan from right because right side is in descending order

        int j = arr.length - 1;
        while (arr[j] <= arr[i]) {
            j--; // find first element greater than arr[i]
        }

        // Swap arr[i] with this just-greater element
        // This ensures minimal increase
        char temp = arr[i];
        arr[i] = arr[j];
        arr[j] = temp;

        // ---------------- STEP 3 ----------------
        // IMPORTANT:
        // Right side (i+1 to end) was in descending order
        // After swap, it's still not the smallest arrangement

        // To make the number just slightly bigger (not too big),
        // we need to make the right part as SMALL as possible

        // Since it's currently descending → reverse it to make ascending
        reverse(arr, i + 1, arr.length - 1);

        // ---------------- FINAL STEP ----------------
        // Convert back to number
        long num = Long.parseLong(new String(arr));

        // Check 32-bit integer limit
        return num > Integer.MAX_VALUE ? -1 : (int) num;
    }

    private void reverse(char[] arr, int l, int r) {
        // Standard reverse of array between l and r
        // Used to convert descending → ascending
        while (l < r) {
            char temp = arr[l];
            arr[l] = arr[r];
            arr[r] = temp;
            l++;
            r--;
        }
    }
}
Why reversing the suffix is correct. After step 1, everything to the right of i is non-increasing (descending). After swapping arr[i] with the next-larger digit from the right, the suffix is still descending — so reversing it yields the lexicographically smallest tail, i.e. the smallest possible number strictly larger than n for this choice of prefix up through index i.

Complexity

  • Naive (permutations + min long): Time O(d! · d) — same leaf count as before, each leaf parses a string of length d. Extra space O(1) beyond the recursion stack (O(d)) and the char buffer; no O(d!) list.
  • ✅ Optimal (next permutation): Time O(d), Space O(d) for the char array — find pivot, swap, reverse suffix, then parse once and compare to Integer.MAX_VALUE.
