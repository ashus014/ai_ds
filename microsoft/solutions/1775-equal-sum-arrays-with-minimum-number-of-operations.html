1775. Equal Sum Arrays With Minimum Number of Operations

You are given two arrays of dice values (each in 1..6). One operation lets you change any single element to any value in 1..6. Return the minimum number of operations to make sum(nums1) == sum(nums2), or -1 if impossible.

Microsoft · Medium

Summary

Hint: Let diff = sum(big) - sum(small) with big being whichever array has the larger sum. Each operation can change diff by at most:

  • shrink an element of big from x down to 1 ⇒ removes up to x - 1 from diff
  • grow an element of small from y up to 6 ⇒ removes up to 6 - y from diff

Since each operation has its own "max gain", a greedy that always burns the move with the largest possible reduction is optimal. Sort both arrays and use two pointers: largest of big from the right, smallest of small from the left.

Answer

✅ Greedy — always take the move with the biggest reduction

Step by step:

  • Compute both sums and ensure nums1 is the bigger one (swap by recursing if not).
  • Quick feasibility check: even if every element of nums1 drops to 1 and every element of nums2 shoots to 6, we need n1 · 1 ≤ n2 · 6. Otherwise return -1.
  • Sort both arrays. Walk i down from the end of nums1 (biggest values first — biggest "shrink potential") and j up from the start of nums2 (smallest values first — biggest "grow potential").
  • At each step, compare nums1[i] - 1 vs 6 - nums2[j] and burn the larger one. Stop when diff <= 0.
✅ Solution.java — greedy + two pointers 
class Solution {
    public int minOperations(int[] nums1, int[] nums2) {
        int sum1 = 0, sum2 = 0;

        // calculate sums
        for (int x : nums1) sum1 += x;
        for (int x : nums2) sum2 += x;

        // always keep nums1 as the one with larger sum
        if (sum1 < sum2) {
            return minOperations(nums2, nums1);
        }

        // quick impossible check
        if (nums1.length * 1 > nums2.length * 6) return -1;

        Arrays.sort(nums1);
        Arrays.sort(nums2);

        int i = nums1.length - 1; // largest in nums1
        int j = 0;               // smallest in nums2

        int diff = sum1 - sum2;
        int ops = 0;

        while (diff > 0) {

            int reduce1 = 0;
            if (i >= 0) {
                reduce1 = nums1[i] - 1;
            }

            int increase2 = 0;
            if (j < nums2.length) {
                increase2 = 6 - nums2[j];
            }

            // if no moves possible → impossible
            if (reduce1 == 0 && increase2 == 0) return -1;

            // pick the better move
            if (reduce1 >= increase2) {
                diff -= reduce1;
                i--;
            } else {
                diff -= increase2;
                j++;
            }

            ops++;
        }

        return ops;
    }
}
Why greedy is optimal here. Every operation reduces diff by an amount that depends only on the element it touches — not on the order. So the minimum number of ops to bring diff <= 0 is achieved by always taking the operation with the largest possible per-op reduction. The two largest piles of "reduction potential" sit at the ends of the sorted arrays: the biggest values of nums1 (each gives x - 1) and the smallest values of nums2 (each gives 6 - y). Sweep both ends, pick the larger one each step.

Why the two impossibility checks?

1. Up-front length check

Even with infinite operations, the smallest possible sum(nums1) is n1 · 1 and the largest possible sum(nums2) is n2 · 6. If n1 > 6 · n2, the bigger array can never come down to meet the smaller array — return -1 immediately, no work done.

Note: nums1.length * 1 > nums2.length * 6 is just n1 > 6·n2 written explicitly to mirror the bounds (min die = 1, max die = 6).

2. In-loop "no moves available"

Once we've consumed every "shrinkable" element of nums1 (everything left is already 1) and every "growable" element of nums2 (everything left is already 6), both reduce1 and increase2 become 0. If diff is still positive at that point, we literally cannot make further progress — return -1. In practice the upfront check usually catches this earlier; the inner check is the safety net.

Walkthrough — nums1 = [1,2,3,4,5,6], nums2 = [1,1,2,2,2,2]

sum1 = 21, sum2 = 10, so nums1 stays as "big". diff = 11.

Sorted: nums1 = [1,2,3,4,5,6], nums2 = [1,1,2,2,2,2]. Pointers i = 5 (value 6), j = 0 (value 1).

step | i nums1[i] | j nums2[j] | reduce1 (x-1) | increase2 (6-y) | choose | diff after | ops -----+------------+------------+---------------+-----------------+-----------------+------------+---- 0 | 5 6 | 0 1 | 5 | 5 | reduce1 (i--) | 11-5=6 | 1 1 | 4 5 | 0 1 | 4 | 5 | increase2 (j++) | 6-5=1 | 2 2 | 4 5 | 1 1 | 4 | 5 | increase2 (j++) | 1-5=-4 | 3 diff <= 0 → stop. answer = 3

Three ops gets us there: e.g. change nums1's 6 → 1 (–5), then nums2's 1 → 6 (–5), then another nums2 1 → 6 (–5). Total diff erased = 15, which already covers the original 11. ✓

Complexity

  • Time: O((n1 + n2) log(n1 + n2)) — dominated by the two sorts. The pointer sweep itself is linear.
  • Space: O(log n) auxiliary for the sorts (Java's primitive Arrays.sort is dual-pivot quicksort). No extra data structures.
