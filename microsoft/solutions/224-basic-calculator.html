224. Basic Calculator

Evaluate an expression of digits, +, -, spaces, and parentheses, returning a 32-bit integer. No × / ÷ here, but the brackets matter — when we hit ( we have to remember the calculation context outside, do the inner one fresh, then merge the result back with the right sign.

Microsoft · Hard

Summary

Hint: Walk the string left → right keeping three things: the running result, the sign attached to the next number (either +1 or −1), and a num being built one digit at a time. On +/−, fold num into result using the current sign, then update sign and reset num. On (, push (result, sign) and start a fresh sub-expression. On ), finish the sub-expression and combine it with the saved outer result and outer sign.

Approach & intention — in plain English

What we are really doing: we are evaluating left to right and treating every number as a signed contribution to result. The sign variable is "the sign that belongs to the next number we are reading." That is why we apply the sign at the moment we close the number (when we see another operator, a ), or end of string), not at the moment we see the operator.

State we maintain

  • result — running total of everything we have committed so far at the current bracket level.
  • sign — +1 or −1, the sign of the number we are currently reading.
  • num — the integer being built digit by digit (e.g. 1 → 12 → 123).
  • stack — for every ( we push two ints: (outerResult, outerSign), so we can resume after the ).

Per character

  • digit → keep building num = num * 10 + (ch - '0').
  • + → commit current number: result += sign * num, then num = 0, sign = +1.
  • - → same commit, then sign = -1.
  • ( → save outer context: push result, push sign, then start fresh: result = 0, sign = +1.
  • ) → commit the last number inside the bracket. Pop prevSign, then prevResult. Merge: result = prevResult + prevSign * result.
  • space → skipped naturally because none of the branches match.

Why we push both result and sign at (

The bracket might be subtracted from the outer context, like 3 - (1 + 2). If we forget the sign in front of the bracket, we cannot flip the inside correctly. Saving (outerResult, outerSign) means the merge step prevResult + prevSign * result always knows whether the inner sub-expression should be added or subtracted.

Worked example: "1 + (4 - (2 + 1))"

char num sign result stack note '1' 1 +1 0 [] '+' 0 +1 1 [] commit 1 '(' 0 +1 0 [1, +1] push outer, reset '4' 4 +1 0 [1, +1] '-' 0 -1 4 [1, +1] commit 4 '(' 0 +1 0 [1, +1, 4, -1] push outer (4, -1), reset '2' 2 +1 0 [1, +1, 4, -1] '+' 0 +1 2 [1, +1, 4, -1] commit 2 '1' 1 +1 2 [1, +1, 4, -1] ')' 0 +1 3 [1, +1] commit 1 → 3; pop (-1, 4) result = 4 + (-1)*3 = 1 ')' 0 +1 1 [] (no last commit, nothing to add) pop (+1, 1) result = 1 + (+1)*1 = 2 end 2 final answer

The two pop steps recover the bracket context exactly the way pen-and-paper math would: first finish what is inside, then bring it back into the outer expression with the correct sign.

Answer

✅ Stack — outer (result, sign) per bracket

✅ Solution.java — sign + stack 
class Solution {
    public int calculate(String s) {
        int result = 0;
        int sign = 1;  // +1 or -1
        int num = 0;

        Stack<Integer> stack = new Stack<>();

        for (char ch : s.toCharArray()) {

            if (Character.isDigit(ch)) {
                num = num * 10 + (ch - '0');
            }
            else if (ch == '+') {
                result += sign * num;
                num = 0;
                sign = 1;
            }
            else if (ch == '-') {
                result += sign * num;
                num = 0;
                sign = -1;
            }
            else if (ch == '(') {
                // save current state
                stack.push(result);
                stack.push(sign);

                // reset for new expression
                result = 0;
                sign = 1;
            }
            else if (ch == ')') {
                result += sign * num;
                num = 0;

                int prevSign = stack.pop();
                int prevResult = stack.pop();

                result = prevResult + prevSign * result;
            }
        }

        // last number
        result += sign * num;

        return result;
    }
}
Why the final result += sign * num; outside the loop. Numbers are only committed when we see the next operator (or a )). The very last number in the string has nothing after it — without that line, an input like "1+2" would return 1 instead of 3.

Complexity

  • Time: O(n) — each character is touched once.
  • Space: O(d) where d is the deepest nesting of parentheses; we push two ints per (. Worst case O(n) for fully nested input like "((((1))))".
