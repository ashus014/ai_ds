127. Word Ladder

Given two words beginWord, endWord, and a dictionary wordList, return the length of the shortest transformation sequence such that each adjacent pair differs by exactly one letter and every intermediate word is in the dictionary. Return 0 if no such sequence exists. The length counts both endpoints.

Microsoft · Hard

Approach in simple words

Picture every word as a dot, and draw a line between any two dots that differ by exactly one letter. Now the question becomes: what is the shortest hop count from the beginWord dot to the endWord dot?

"Shortest path on an unweighted graph" is the textbook signal for BFS. BFS explores in concentric rings: first all words 1 hop away, then all words 2 hops away, and so on. The first ring that contains endWord is the answer.

Two practical tricks make this fast:

  1. Generate neighbors on the fly. Don't precompute the whole graph. For each word in the queue, try replacing every position with every letter a–z. If the candidate is in the dictionary, it's a neighbor. That's only 26 × L tries per word.
  2. Use the dictionary set as the "visited" tracker. The moment you enqueue a word, remove() it from wordSet. It can never be reached again on a shorter path (BFS guarantees that), and now you can't accidentally re-enqueue it.

To count "how many words in the chain so far," BFS in level order: snapshot queue.size() at the start of each round, drain exactly that many words, then bump level. That round's words are all at the same hop distance from beginWord.

Summary

Hint: Treat each word as a graph node, edges = "differs by one letter," and BFS from beginWord. Generate neighbors by mutating each position to every letter a–z and checking the dictionary set. Strip enqueued words out of the set so we never revisit them. Process the queue level-by-level, returning the level at which we pull endWord off.

Answer

✅ BFS — generate neighbors by mutating one letter at a time

✅ Solution.java — BFS 
import java.util.*;

class Solution {
    public int ladderLength(String beginWord, String endWord, List<String> wordList) {

        // store all words for fast lookup
        Set<String> wordSet = new HashSet<>(wordList);

        // if endWord not present → impossible
        if (!wordSet.contains(endWord)) {
            return 0;
        }

        // BFS queue
        Queue<String> queue = new LinkedList<>();
        queue.offer(beginWord);

        int level = 1; // starting word counts as 1

        while (!queue.isEmpty()) {

            int size = queue.size(); // process level by level

            for (int i = 0; i < size; i++) {
                String word = queue.poll();

                // if reached endWord → return steps
                if (word.equals(endWord)) {
                    return level;
                }

                // try all possible transformations
                char[] arr = word.toCharArray();

                for (int j = 0; j < arr.length; j++) {

                    char original = arr[j];

                    // replace with 'a' to 'z'
                    for (char ch = 'a'; ch <= 'z'; ch++) {

                        arr[j] = ch;
                        String newWord = new String(arr);

                        // if valid word in dictionary
                        if (wordSet.contains(newWord)) {
                            queue.offer(newWord);

                            // mark visited (remove to avoid reuse)
                            wordSet.remove(newWord);
                        }
                    }

                    // restore original character
                    arr[j] = original;
                }
            }

            level++; // move to next level
        }

        return 0; // no path found
    }
}
Why "remove from set" is a valid visited-marker. BFS visits nodes in non-decreasing distance order. The very first time we reach a word is via the shortest possible path; any later route to the same word would only be longer (or equal). So once enqueued, that word can be safely deleted from wordSet — re-enqueuing it could never improve the answer, only waste work and risk infinite loops.
Why the queue.size() snapshot. We're growing the queue while we drain it, so we capture size once at the top of each level. We then poll exactly that many words — those are all at the current level. Anything offered during this round goes to level + 1. This is the standard BFS-by-levels pattern.

Walkthrough — begin = "hit", end = "cog", dict = ["hot","dot","dog","lot","log","cog"]

Expected output: 5, via the chain hit → hot → dot → dog → cog (5 words counted).

level | queue (state when level starts) | what we generate this round ------+----------------------------------------+-------------------------------------------------- 1 | [hit] | "hit": mutate each slot. Only "hot" exists in | | the dictionary. → enqueue "hot", remove from set. set after = {dot, dog, lot, log, cog} ------+----------------------------------------+-------------------------------------------------- 2 | [hot] | "hot" → neighbors in set: "dot", "lot". enqueue both, remove. set after = {dog, log, cog} ------+----------------------------------------+-------------------------------------------------- 3 | [dot, lot] | "dot" → "dog" (in set). enqueue, remove. | | "lot" → "log" (in set). enqueue, remove. set after = {cog} ------+----------------------------------------+-------------------------------------------------- 4 | [dog, log] | "dog" → "cog" (in set!). enqueue, remove. | | "log" → "cog" already gone — nothing new. set after = {} ------+----------------------------------------+-------------------------------------------------- 5 | [cog] | poll "cog" → equals endWord → return level = 5 ✓

Notice how at level 4 both dog and log could lead to cog, but BFS only enqueues it once because the first sighting removes it from the set — saving a duplicate visit at no cost to correctness.

Edge cases

  • endWord not in dictionary. Caught upfront — return 0. Without this guard the BFS would still terminate correctly but waste time exploring the graph.
  • beginWord == endWord. Pulled from queue immediately at level 1, returns 1. (Per LeetCode spec, beginWord is not required to be in the dictionary; it counts as the first word in the chain.)
  • No reachable path. Queue eventually drains, fall through to return 0.
  • beginWord in dictionary. Harmless — we never re-enqueue it anyway because we don't push the seed twice. (If it had self-mutation back to itself, the equality check at "a" → "a" still produces the same word, but it's already been pulled and processed; remove just no-ops on the next sighting because the set lookup misses.)
  • Length-1 words. Inner loop runs once per word, mutating that single slot through 26 letters — works.

Complexity

  • Time: O(N · L · 26) = O(N · L), where N = words in the dictionary and L = word length. Each word is dequeued at most once (we strip it from the set on enqueue). For each, we try L · 26 mutations and a hash lookup on each. Building the candidate string is another O(L), so a tighter bound is O(N · L² · 26).
  • Space: O(N · L) for the dictionary set and the queue (in the worst case the queue can hold a full BFS frontier).
Faster variant — bidirectional BFS. When the dictionary is large, BFS from both beginWord and endWord simultaneously, always expanding the smaller frontier. They meet in the middle, and the total work shrinks roughly from bd to 2·bd/2. Same algorithmic complexity class but a big constant-factor win for hard test cases.
Alternate neighbor lookup — generic patterns. Pre-build a map keyed by patterns like "h*t" → ["hot","hat",…], "*ot" → ["hot","dot","lot",…]. Then a word's neighbors are the union of buckets for each of its L patterns — no 26 factor. Trade-off: a one-time O(N · L²) preprocess for cheaper per-word neighbor lookups during BFS.
