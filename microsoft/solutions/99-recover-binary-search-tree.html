99. Recover Binary Search Tree

Two nodes of a BST were swapped by mistake. Restore the tree without changing its structure. Use the BST property: an inorder traversal must produce a strictly increasing sequence — when it does not, the offenders are exactly the two swapped nodes.

Microsoft · Medium

Summary

Hint: Do an inorder traversal and collect the nodes in a list. Walk adjacent pairs (list[i], list[i+1]) looking for list[i].val > list[i+1].val — call those violations. The two swapped nodes are: the first entry of the first violation, and the second entry of the last violation. Swap their val fields, done.

Answers

✅ Inorder list + swap two values

✅ Solution.java — inorder list 
/**
 * Definition for a binary tree node.
 * public class TreeNode {
 *     int val;
 *     TreeNode left;
 *     TreeNode right;
 *     TreeNode() {}
 *     TreeNode(int val) { this.val = val; }
 *     TreeNode(int val, TreeNode left, TreeNode right) {
 *         this.val = val;
 *         this.left = left;
 *         this.right = right;
 *     }
 * }
 */
class Solution {
    public void recoverTree(TreeNode root) {
        List<TreeNode> list = new ArrayList<>();

        // step 1: inorder
        inorder(root, list);

        // step 2: find two wrong nodes
        TreeNode first = null, second = null;

        for (int i = 0; i < list.size() - 1; i++) {
            if (list.get(i).val > list.get(i + 1).val) {
                if (first == null) {
                    first = list.get(i);
                }
                second = list.get(i + 1);
            }
        }

        // step 3: swap values
        int temp = first.val;
        first.val = second.val;
        second.val = temp;
    }

    private void inorder(TreeNode node, List<TreeNode> list) {
        if (node == null) return;

        inorder(node.left, list);
        list.add(node);
        inorder(node.right, list);
    }
}

Optimal — O(n) time, O(1) extra space*

Same idea, but instead of materializing the full inorder list, we keep a single prev pointer and check prev.val > node.val at each visit. first still locks once on the first violation; second still overwrites on every violation (so it ends on the last one). *ignoring the recursion stack O(h).

Solution.java — inorder + prev pointer 
class Solution {
    TreeNode first = null;
    TreeNode second = null;
    TreeNode prev = null;

    public void recoverTree(TreeNode root) {
        inorder(root);

        // swap
        int temp = first.val;
        first.val = second.val;
        second.val = temp;
    }

    private void inorder(TreeNode node) {
        if (node == null) return;

        inorder(node.left);

        // violation check
        if (prev != null && prev.val > node.val) {
            if (first == null) {
                first = prev;
            }
            second = node;
        }

        prev = node;

        inorder(node.right);
    }
}
Why prev is enough. The list version checks neighbors via list[i] and list[i+1]. In a recursive inorder, the "previous node" in inorder order is just whatever node finished its visit right before this one — so we cache it in a field and compare. No extra array, same logic. To strip the recursion stack as well, swap the recursion for Morris inorder for true O(1) total auxiliary space.

Why second = list.get(i + 1); sits outside the inner if

Easy way to remember (don't overthink it): there are two possible situations. The code has to handle both with the same loop.

✅ Case 1 — swapped nodes are adjacent in inorder

Example:

Original inorder: 1 2 3 4 After swap: 1 3 2 4 ^ ^ swapped (3 and 2)

Only one violation:

3 > 2 ❌

So we set first = 3 (the bigger one in the violation) and second = 2 (the smaller one right after). Both happen on the same iteration.

✅ Case 2 — swapped nodes are far apart

Example: swap 1 and 3 in 1 2 3 4:

Original inorder: 1 2 3 4 After swap: 3 2 1 4 ^ ^ swapped (1 and 3)

Now two violations show up:

3 > 2 ❌ first violation 2 > 1 ❌ second violation

From the first violation we want to grab first = 3. From the second violation we want to grab second = 1.

That is exactly what the loop does

  • first is set only once — guarded by if (first == null) — so it locks in on the first violation.
  • second is set every time a violation is seen — no if guard — so by the end of the loop it holds list.get(i + 1) from the last violation.

This handles both cases with one pass:

  • Adjacent (Case 1): only one violation fires; both first and second are set in that single iteration.
  • Non-adjacent (Case 2): the first violation sets first (and tentatively second); the second violation overwrites second with the correct one.
If you put second inside the same if (first == null), Case 2 breaks — second would freeze on 2 from the first violation instead of moving on to 1 from the second one. You'd "fix" the tree by swapping 3 ↔ 2, which is wrong.

Complexity

  • Time: O(n) for both variants — one inorder traversal (the list version also does a linear scan after, still O(n)).
  • Space (list version): O(n) for the inorder list, plus O(h) recursion stack.
  • Space (prev-pointer version): O(1) extra besides the O(h) recursion stack. Morris inorder strips that too for true O(1) total auxiliary space.
