1647. Minimum Deletions to Make Character Frequencies Unique

Given a lowercase string s, return the minimum number of character deletions so that no two distinct characters share the same frequency. Frequency 0 doesn't count as a "shared" frequency — once a character is fully deleted, it's gone from the set.

Microsoft · Medium

Summary

Hint: Count frequencies for the 26 letters. Walk those counts and, for each one, drop it down (one delete at a time) until it lands on a value no one else has used yet — or hits 0. Total drops = answer.

Answers

✅ Frequency array + HashSet of "used" counts

Count how many times each of the 26 letters appears. Then, for every count, keep decrementing it (and adding to deletions) while that value is already taken in the set; once it lands on a free value (or 0), record it.

✅ Solution.java — freq[] + HashSet 
class Solution {
    public int minDeletions(String s) {
        int[] freq = new int[26];

        // count frequencies
        for (char c : s.toCharArray()) {
            freq[c - 'a']++;
        }

        Set<Integer> used = new HashSet<>();
        int deletions = 0;

        for (int f : freq) {
            while (f > 0 && used.contains(f)) {
                f--;          // delete one character
                deletions++;
            }
            used.add(f);
        }

        return deletions;
    }
}
Why f > 0 in the while? If a character gets squeezed all the way down to 0 we should stop. Adding 0 to the set is fine — multiple characters can have frequency 0 (i.e. be fully deleted) without violating the "unique frequencies" rule, because they're not in the string anymore. The f > 0 guard prevents us from incorrectly deleting more characters once we've hit zero.

Follow-up — without a HashSet (sort + cascade)

If the interviewer says "do it without a hashset / extra map", sort the 26 frequencies ascending and walk from the second-largest down. The largest stays as-is; every smaller (or equal) neighbour just below it gets pushed down to freq[i + 1] - 1 (and never below 0). The cumulative drop is the answer.

Solution.java — sort + cascade (no HashSet) 
class Solution {
    public int minDeletions(String s) {
        int[] freq = new int[26];

        // count frequencies
        for (char ch : s.toCharArray()) {
            freq[ch - 'a']++;
        }

        Arrays.sort(freq);

        int result = 0;

        // iterate from second largest (like i = 24)
        for (int i = 24; i >= 0 && freq[i] > 0; i--) {
            if (freq[i] >= freq[i + 1]) {
                int prev = freq[i];

                int newFreq = freq[i + 1] - 1;
                if (newFreq < 0) newFreq = 0;

                freq[i] = newFreq;

                result += (prev - freq[i]);
            }
        }

        return result;
    }
}
Why sort + walk from the right? After sorting ascending, freq[25] is the biggest count and is automatically "safe" — no one else is bigger, so nothing needs to drop on its account. Walking i = 24 … 0, each entry only has to undercut its right neighbour by at least 1 (clamped at 0). Greedy is optimal here because keeping the bigger value untouched and pushing the smaller one down is always at least as cheap as the reverse.

Walkthrough — "aaabbbcc"

Frequencies: a=3, b=3, c=2. The two 3s collide.

HashSet version

freq = [3, 3, 2, 0, 0, ...] (a=3, b=3, c=2) used = {}, deletions = 0 f = 3 not in used → used = {3} f = 3 in used → f=2, deletions=1 2 not in used → used = {3, 2} f = 2 in used → f=1, deletions=2 1 not in used → used = {3, 2, 1} f = 0 while-guard f > 0 fails → used.add(0) (harmless) ... (remaining 0s same as above) answer = 2 ✓

Sort + cascade version

freq = [3, 3, 2, 0, 0, ..., 0] (a=3, b=3, c=2) after sort asc = [0, 0, ..., 0, 2, 3, 3] ^ ^ ^ 23 24 25 i = 24 (value 3): freq[24]=3 >= freq[25]=3 newFreq = 3 - 1 = 2 → freq[24] = 2 result += 3 - 2 = 1 (result = 1) i = 23 (value 2): freq[23]=2 >= freq[24]=2 newFreq = 2 - 1 = 1 → freq[23] = 1 result += 2 - 1 = 1 (result = 2) i = 22 (value 0): loop guard freq[i] > 0 fails → stop result = 2 ✓

Both approaches agree on 2. The cascade just walks neighbours; the HashSet version walks "the next free integer below my current value."

Complexity

Approach Time Space
✅ Frequency array + HashSet O(n + Σ) — one scan of the string (n), then for each of the 26 letters at most O(max freq) decrements; total decrements bounded by n in the worst case O(26) for the freq array + O(26) for the set ⇒ O(1) auxiliary
Follow-up — sort + cascade (no HashSet) O(n + 26 log 26) ⇒ effectively O(n) O(1) — sorts a fixed-size array of 26 in place

Both are linear in n; the cascade version is the one to reach for when the interviewer rules out auxiliary hashing.
