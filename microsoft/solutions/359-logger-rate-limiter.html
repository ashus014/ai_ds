359. Logger Rate Limiter

For each (timestamp, message) stream entry, decide whether the message is allowed to print right now — given that any unique message can fire at most once per 10-second window.

Microsoft · Easy

Summary

Hint: For every message, remember the earliest timestamp at which it is allowed to be printed again. Store that in a hashmap message → nextAllowedTs. On a new call: if timestamp < nextAllowedTs, reject; otherwise accept and update nextAllowedTs = timestamp + 10.

Answer

✅ HashMap of message → next allowed timestamp

✅ Logger.java — HashMap of next-allowed-ts 
class Logger {
    private Map<String, Integer> ts = new HashMap<>();

    public Logger() {
    }

    public boolean shouldPrintMessage(int timestamp, String message) {
        int t = ts.getOrDefault(message, 0);
        if (timestamp < t) {
            return false;
        }
        ts.put(message, timestamp + 10);
        return true;
    }
}

/**
 * Your Logger object will be instantiated and called as such:
 * Logger obj = new Logger();
 * boolean param_1 = obj.shouldPrintMessage(timestamp,message);
 */
Why store timestamp + 10 (next allowed) instead of just the last printed timestamp? Both work, but storing the next allowed ts lets the check stay a single direct comparison: if (timestamp < t) return false;. No subtraction, no off-by-one ambiguity around the 10-second boundary.

Walkthrough — given example

Calls (in order): (1,"foo"), (2,"bar"), (3,"foo"), (8,"bar"), (10,"foo"), (11,"foo").

call map BEFORE decision map AFTER (1, "foo") {} 1 >= 0 → true {"foo": 11} (2, "bar") {"foo":11} 2 >= 0 → true {"foo":11, "bar":12} (3, "foo") {"foo":11, "bar":12} 3 < 11 → false (unchanged) (8, "bar") {"foo":11, "bar":12} 8 < 12 → false (unchanged) (10, "foo") {"foo":11, "bar":12} 10 < 11 → false (unchanged) (11, "foo") {"foo":11, "bar":12} 11 >= 11 → true {"foo":21, "bar":12}

Output: [null, true, true, false, false, false, true] ✓

Edge cases worth noting

  • Boundary equality. The window is inclusive at t + 10 — a message printed at t = 1 may print again at t = 11 but not at t = 10. The check timestamp < t (strict) gets this right.
  • Same timestamp, repeated message. First call wins; later calls in the same timestamp see timestamp < nextAllowedTs and return false.
  • Same timestamp, different messages. Each is tracked independently in the map, so they don't block each other.
  • First-ever call. getOrDefault(message, 0) returns 0 and any non-negative timestamp passes — no special-case branch needed.

Complexity

  • Time per call: O(1) amortized — one hashmap lookup + at most one insert.
  • Space: O(U) where U is the number of unique messages ever seen. The map keeps growing as new messages arrive; entries are never evicted.
Memory note (interview follow-up). If the interviewer pushes on memory growth — "what if the stream runs forever?" — you can switch to a queue + set design: a Queue<int[]> of (timestamp, message) records the last 10 seconds, and a HashSet<String> mirrors what's currently in the queue. On every call, pop expired entries from the queue head (and their string from the set), then check the set. That trades a tiny bit of per-call work for bounded memory of "messages within the last 10 seconds."
