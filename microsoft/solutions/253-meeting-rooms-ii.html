253. Meeting Rooms II

Given a list of meeting [start, end) intervals, return the minimum number of conference rooms required so that no two meetings collide. Touching at a boundary — e.g. (0,8) and (8,10) — does not count as a conflict.

Microsoft · Medium

Summary

Hint: The answer is the maximum number of meetings that overlap at any single instant. To compute this in O(n log n): sort meetings by start time, then sweep them in order while keeping a min-heap of the end times of currently-occupied rooms. For each new meeting, if the earliest-ending room is already free (its end time ≤ this meeting's start), reuse that room (poll it); always push this meeting's end time. The heap's final size is the answer.

Answer

✅ Sort by start + min-heap of end times

Step by step:

  • Sort intervals by start ascending — we'll process meetings in the order they begin.
  • Maintain a min-heap whose contents are the end times of every room currently in use. heap.peek() tells us the earliest a room will become free.
  • For each meeting, if heap.peek() <= meeting.start, that room finished in time — poll() to free it (we'll immediately reassign it). Then add(meeting.end) to mark whichever room (reused or brand-new) is now busy until then.
  • The size of the heap at the end equals how many distinct rooms we had in flight — the answer.
✅ Solution.java — sort + min-heap 
/**
 * Definition of Interval:
 * public class Interval {
 *     public int start, end;
 *     public Interval(int start, int end) {
 *         this.start = start;
 *         this.end = end;
 *     }
 * }
 */

class Solution {
    public int minMeetingRooms(List<Interval> intervals) {
        if (intervals == null || intervals.size() == 0) return 0;

        // sort by start time
        intervals.sort((a, b) -> a.start - b.start);

        PriorityQueue<Integer> minHeap = new PriorityQueue<>();

        for (Interval meeting : intervals) {
            // if room is free
            if (!minHeap.isEmpty() && minHeap.peek() <= meeting.start) {
                minHeap.poll();
            }

            // allocate room
            minHeap.add(meeting.end);
        }

        return minHeap.size();
    }
}
Why the heap's final size is the answer. Every add represents "this room is now booked." Every poll represents "an existing room came free in time, reuse it." So the heap only grows when we genuinely had to spin up a new room. After processing every meeting, the heap holds one entry per room ever opened — equal to the peak concurrent meetings, which is exactly the rooms required.
Why <= and not <. The problem treats (0, 8) ending and (8, 10) starting as not a conflict — the meeting that ends at 8 vacates the room exactly when the next meeting starts. peek() <= meeting.start captures that "boundary touch" correctly. If the problem treated (0, 8) / (8, 10) as overlapping, you'd switch to a strict <.

Walkthrough — intervals = [(0, 40), (5, 10), (15, 20)]

Already sorted by start. The expected answer is 2 (room 1 holds (0,40); room 2 holds (5,10) then (15,20)).

step | meeting | heap before | peek <= start? | action | heap after -----+-----------+-------------+-----------------+------------------------------+------------ 1 | (0, 40) | [] | n/a (empty) | add 40 | [40] 2 | (5, 10) | [40] | 40 <= 5 → no | add 10 | [10, 40] 3 | (15, 20) | [10, 40] | 10 <= 15 → yes | poll 10, then add 20 | [20, 40] heap size at the end = 2 ✓

Notice how (15,20) reused the room that (5,10) just vacated, so the heap stayed at size 2 instead of climbing to 3.

Edge cases

  • Empty input. The upfront null/size==0 guard returns 0 directly.
  • One meeting. Single add, heap size 1, answer 1.
  • Boundary touch — (0,8) then (8,10). After processing (0,8), heap is [8]. For (8,10): 8 <= 8 → poll, then add 10. Heap stays size 1. ✓ (no new room).
  • All meetings overlap. No poll ever fires, heap grows to size n — answer n.
  • Identical start times. Sort tie-breaker doesn't matter; whichever order we visit them, both will end up booking rooms (the heap is empty for the first, and the second sees peek() = first.end > common start, so no reuse).

Complexity

  • Time: O(n log n) — the sort dominates; the heap loop is n iterations of O(log n) each.
  • Space: O(n) — the heap can hold up to one entry per concurrently active room (worst case n if all meetings overlap).
Equivalent classic alternative — two-pointer sweep. Instead of a heap, build two sorted arrays: all start times and all end times, each sorted ascending. Walk both with pointers s, e; if start[s] < end[e] a meeting begins before any ends — increment a rooms counter and advance s. Otherwise a room frees up — decrement (or just advance e while tracking max). The maximum value of rooms during the sweep is the answer. Same O(n log n) from the two sorts, but no heap needed.
