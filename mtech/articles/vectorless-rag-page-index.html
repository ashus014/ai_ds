Vectorless RAG & Page Index

Hierarchical, reasoning-based retrieval vs. traditional vector RAG

📅 20 March 2026

Summary

The video presents an in-depth explanation of Vectorless RAG (Retrieval-Augmented Generation), a novel approach to document retrieval and question answering that contrasts with traditional vector-based RAG systems. The presenter outlines the challenges in existing vector RAG techniques and introduces Page Index, a vectorless reasoning-based alternative that leverages hierarchical data structures and large language models (LLMs) for improved document understanding and retrieval.

Core concepts and workflow

Traditional RAG and its challenges

  • RAG — Retrieval-Augmented Generation: many documents are chunked, embedded into vectors, stored in a vector database, and retrieved to ground generation.
  • The user submits a query; the system answers by combining retrieved chunks and the query.

Problems with traditional vector RAG:

  • Limited context window — LLMs have finite tokens; very large documents (e.g. 100+ pages) exceed practical windows.
  • Output quality degradation — stuffing too much context can cause hallucination or unfocused answers.
  • High cost — embeddings and long prompts increase compute and API cost.
  • Chunking issues — fixed-size chunks (e.g. 500 words) can split semantics arbitrarily.
  • Semantic matching dependency — retrieval depends on embedding similarity; vague or differently worded queries hurt recall.
  • Reference problems — legal and structured docs use cross-references; naive chunking can drop that structure.

Traditional vector RAG pipeline

StepDescription
Indexing Chunk documents (by page, paragraph, or fixed word count); embed with models such as OpenAI text-embedding-3; store in vector DBs (e.g. Pinecone, Chroma, PGVector).
Query Embed the query; similarity search for relevant chunks; pass chunks + query to the LLM to generate the answer.

Vectorless RAG (Page Index)

  • Vectorless RAG does not use vector embeddings or vector similarity search.
  • It builds a hierarchical tree (TOC-style) using LLM reasoning:
    • Parse the document into semantic nodes (chapters, sections, scenarios, plot points, etc.).
    • Each node: title, unique ID (pointer into the source), summary, child nodes.
  • Mimics how people use a table of contents rather than linearly scanning every page.

Indexing phase (vectorless)

  • No chunking for embedding; no vector index.
  • LLM identifies structure (e.g. for a film like Sholay: scenes, characters, transitions).
  • Produces semantic segmentation, summaries, and a hierarchical index.
  • Mitigates bad splits, weak chunk semantics, and lost cross-references when the tree preserves links.

Query phase (vectorless)

  • LLM traverses the TOC tree with reasoning (e.g. breadth-first exploration).
  • Uses node summaries and metadata to keep relevant branches and prune irrelevant ones.
  • Fetches only the needed summaries and linked spans from the original document.
  • Passes focused context to the LLM for generation — less noise than dumping similar chunks only.

Technical notes

  • Vector RAG: embedding models (e.g. OpenAI text-embedding-3 small/large) + stores such as Pinecone, Chroma, Milvus, PGVector.
  • Vectorless RAG: relies on LLM reasoning to build and query the index.
  • Tree nodes typically hold: title, node ID, summary, children.
  • Answering = guided tree traversal, then grounded generation on selected spans.
  • Aligned with human reading: jump to chapters/sections instead of uniform scanning.

Trade-offs and status

  • Vectorless approaches can trade speed for accuracy: multiple LLM steps for traversal and filtering are expensive.
  • Still relatively new; promising for contracts, books, and other large, reference-heavy corpora.
  • Speaker notes migration from vector RAG to vectorless in some projects; optional coding tutorials.
  • Benefits from stronger LLM reasoning and document-understanding capabilities over time.

Key insights

  • Vectorless RAG avoids embedding + similarity search and some of their failure modes.
  • Hierarchical semantic indexing (page index) supports focused, context-aware retrieval.
  • Especially relevant for large documents with cross-references and semantic dependencies.
  • Main downside: compute and latency from reasoning-heavy traversal.
  • Conceptually close to how humans navigate structured texts.

Conclusion

The video contrasts limitations of vector-based RAG with vectorless RAG as a reasoning-driven, hierarchical alternative. A semantic tree plus LLM-guided navigation addresses chunking noise, context limits, and brittle lexical/semantic retrieval. It costs more per query in reasoning steps but is a meaningful direction for complex, large-scale document QA.
