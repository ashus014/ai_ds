1268. Search Suggestions System

LeetCode 75 #71 — After sorting products, each typed prefix has a contiguous block of matches in lexicographic order. Arrays.binarySearch finds where that block starts; then take at most three strings that still startsWith(prefix).

LeetCode 75 / Trie

Hint

When binarySearch returns a negative value, decode the insertion point as -index - 1. Because the array is sorted, once startsWith(prefix) fails you can stop the inner loop early.

Answers

✅ Sort + binary search

No explicit trie: sorting plus binary search on the growing prefix is enough for the constraints.

Solution.java — sort + binarySearch 
class Solution {
    public List<List<String>> suggestedProducts(String[] products, String searchWord) {

        // Sort products lexicographically
        // So similar prefixes stay together (VERY IMPORTANT)
        Arrays.sort(products);

        List<List<String>> result = new ArrayList<>();

        String prefix = "";

        // Build prefix one character at a time
        for (char c : searchWord.toCharArray()) {

            prefix += c; // e.g. "m" → "mo" → "mou"

            List<String> suggestions = new ArrayList<>();

            // Binary search to find where this prefix would fit
            int index = Arrays.binarySearch(products, prefix);

            // If prefix is not found exactly,
            // binarySearch returns negative insertion point
            if (index < 0) {
                index = -index - 1; // convert to insertion index
            }

            // From that index, check next 3 products
            // (because we only need top 3 suggestions)
            for (int i = index; i < Math.min(index + 3, products.length); i++) {

                // If product starts with prefix → valid suggestion
                if (products[i].startsWith(prefix)) {
                    suggestions.add(products[i]);
                } else {
                    // Since sorted, if one doesn't match,
                    // no further ones will match → break early
                    break;
                }
            }

            // Add suggestions for this prefix
            result.add(suggestions);
        }

        return result;
    }
}
Sorted array. Sorting keeps all strings that share a prefix in one contiguous stretch, so starting from the insertion index and scanning forward is enough. The section below explains Arrays.binarySearch and why index = -index - 1; recovers that starting index when the prefix is not an exact element.

Why index = -index - 1; matters

What does Arrays.binarySearch() return?

It returns two types of values:

✅ Case 1 — element found: index >= 0 — that is the actual index in the array.

❌ Case 2 — element not found: index < 0 — you get a negative number, not a usable index directly.

What does that negative number mean?

It encodes the insertion point: where this key should go to keep the array sorted. Internally Java uses:

returned_value = -(insertion_index) - 1

So to get the actual insertion index, reverse it:

index = -index - 1;

That is exactly undoing -(insertion) - 1.

Example — prefix = "car" (super important)

products = ["apple", "banana", "cat", "dog"]

Sorted order: apple, banana, cat, dog. String "car" would be inserted between banana and cat, so the insertion index is 2.

binarySearch returns -(2) - 1 = -3. Your code:

if (index < 0) {
    index = -index - 1;
}
// index = -(-3) - 1 = 2  ✓ correct insertion index

Another example — prefix = "zebra"

products = ["apple", "banana", "cat"]

"zebra" goes after everything → insertion index 3 (end). binarySearch returns -(3) - 1 = -4. Then index = -(-4) - 1 = 3.

Why this is useful here

You want where to start checking for products that can still match the current typed prefix. Decoding the insertion index gives that exact starting position; then you walk forward (at most three strings) in sorted order.
