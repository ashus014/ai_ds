Design princple

DRY — Don’t Repeat Yourself
Don’t duplicate logic everywhere. Keep common code in one place so it’s easy to change and maintain.

YAGNI — You Aren’t Gonna Need It
Don’t build extra features “just in case.”
Build only what is needed right now.

KISS — Keep It Simple, Stupid
Prefer simple, readable designs instead of clever, complicated ones.

SOLID Principles

S — Single Responsibility Principle
One class = one main job.
If a class is doing too many things, split it.

O — Open/Closed Principle
Code should be open for extension but closed for modification.
You extend behavior (add new classes), not rewrite existing ones.

L — Liskov Substitution Principle
A child class should be usable anywhere its parent is expected —
without breaking the program.

I — Interface Segregation Principle
Don’t force classes to implement big, unnecessary interfaces.
Create smaller, focused interfaces.

D — Dependency Inversion Principle (not Injection)
High-level modules shouldn’t depend directly on low-level ones — both depend on abstractions (interfaces).
Dependency Injection is one way to achieve this.

design peincples

Singleton
Only one object of a class should ever exist.
Example: one global logger, one DB connection manager.

🔹 Factory
You ask the factory for an object — it decides which exact class to create.
Good when object creation logic gets messy.

🔹 Abstract Factory
A factory of factories.
Used to create families of related objects without knowing their concrete classes.

🔹 Builder
Used when objects have lots of optional fields.
Lets you build an object step-by-step instead of using a huge constructor.

🔹 Adapter
Makes two incompatible classes work together.
Like a charger plug adapter.

🔹 Decorator
Adds extra features without changing existing code.
Wraps an object and enhances it.

Facade
Provides a simple interface to a complex system.
Like a hotel reception desk handling everything for you.

🔹 Composite
Treats a group of objects the same way as a single object.
Useful for tree-like structures (folders, UI components).

🔹 Observer
One object changes → automatically notifies others.
Like YouTube channel notifications to subscribers.

🔹 Command
Wraps an action as an object.
Helps with undo/redo, queues, and logging.

🔹 Strategy
Different algorithms, same goal — switch them at runtime.
Example: different sorting methods.

🔹 State
Behavior changes based on internal state.
Example: Media player → playing / paused/stopped.

🔹 Template Method
Defines the skeleton of an algorithm, but lets subclasses fill steps.
Base class decides structure, child classes customize parts.

how ro answer a LLD interview

How to answer an LLD interview

In LLD interviews, you are given a real-world problem and expected to come up with a solution using object-oriented programming.

Clarify requirements and core use cases
Ask questions to understand what we need to do and the core use cases we need to support.

Identify the entities or objects

Translate entities into classes and come up with a list of attributes

Identify core methods

Define relationships between classes
Draw UML diagrams to represent relationships.

Implement necessary methods
No need to implement all methods — check with the interviewer to understand which methods are important.

Exception handling
Handle errors and edge cases.

Best LLD coding practices

Use meaningful names
Class names, method names, and variable names.

Focus on simplicity and readability

Favor composition over inheritance
Promote flexible and tight coupling.

Use interfaces to define contracts
Enable loose coupling between components.

Strive for modularity and separation of concern

Apply design patterns and design principles

Make it scalable
