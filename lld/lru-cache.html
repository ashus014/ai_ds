LRU Cache

Least Recently Used cache with fixed capacity: evict the least recently used entry when full. Naive ⚠️ uses a list and linear key search; optimised ℹ️ adds a HashMap from key → list index and updateMap() after each reorder.

LLD / Java — 12 April 2026

Summary

Store key–value pairs with cap n. get marks an entry as most recent (front of list). put updates or inserts at the front; when full, drop the tail. The map-backed version avoids scanning for the key but still pays for ArrayList shifts and rebuilding indices.

Answers

Naive approach ⚠️

List of [key, value] pairs; MRU at index 0. get scans, then move-to-front. put scans for an existing key or inserts at front; evict tail when at capacity.

  • Linear scan on every get / put.
  • No auxiliary structure — easy to explain, slow on large capacity.
LRUCache.java — naive ⚠️ 
import java.util.*;

class LRUCache {

    List<int[]> cache;
    int n;

    public LRUCache(int capacity) {
        this.n = capacity;
        this.cache = new ArrayList<>();
    }

    public int get(int key) {

        for (int i = 0; i < cache.size(); i++) {
            if (cache.get(i)[0] == key) {

                int value = cache.get(i)[1];

                // move to front (most recent)
                cache.remove(i);
                cache.add(0, new int[]{key, value});

                return value;
            }
        }

        return -1;
    }

    public void put(int key, int value) {

        // check if key already exists
        for (int i = 0; i < cache.size(); i++) {
            if (cache.get(i)[0] == key) {

                // remove old position
                cache.remove(i);

                // insert at front (most recent)
                cache.add(0, new int[]{key, value});
                return;
            }
        }

        // if capacity full → remove least recently used (last element)
        if (cache.size() == n) {
            cache.remove(cache.size() - 1);
        }

        // insert new at front
        cache.add(0, new int[]{key, value});
    }
}
Naive ⚠️. Every get / put scans the whole list for the key — O(n) time plus list remove/insert. Fine for learning; not for production-scale caches.

Optimised (map + list) ℹ️

Map<Integer, Integer> stores key → index in the list. After remove / add(0, …), call updateMap() to refresh every index. On eviction, remove the tail pair and map.remove(removedKey).

  • O(1) key lookup in the map; list shifts and updateMap() remain O(n) overall.
  • Step toward a classic DLL + hash LRU (O(1) get/put), not a full substitute.
LRUCache.java — optimised ℹ️ 
import java.util.*;

class LRUCache {

    List<int[]> cache;
    Map<Integer, Integer> map; // key → index
    int n;

    public LRUCache(int capacity) {
        this.n = capacity;
        this.cache = new ArrayList<>();
        this.map = new HashMap<>();
    }

    public int get(int key) {

        if (!map.containsKey(key)) return -1;

        int index = map.get(key);
        int value = cache.get(index)[1];

        cache.remove(index);
        cache.add(0, new int[]{key, value});

        updateMap();

        return value;
    }

    public void put(int key, int value) {

        if (map.containsKey(key)) {
            int index = map.get(key);
            cache.remove(index);
        }
        else if (cache.size() == n) {
            int removedKey = cache.get(cache.size() - 1)[0];
            cache.remove(cache.size() - 1);
            map.remove(removedKey);
        }

        cache.add(0, new int[]{key, value});

        updateMap();
    }

    private void updateMap() {
        for (int i = 0; i < cache.size(); i++) {
            map.put(cache.get(i)[0], i);
        }
    }
}
Map-assisted ℹ️. You skip the linear search for the key, but remove(index), insert at 0, and updateMap() still scale linearly with capacity — same asymptotic class as naive, with a better leading constant on lookup.

Complexity

Naive ⚠️

  • get(key), put(key, value): O(n) for scan + list ops.
  • Space: O(capacity).

Map-assisted ℹ️

  • Map lookup O(1); list + updateMap() dominate — still O(n) per operation worst case.
  • Space: O(capacity) for list + map.
